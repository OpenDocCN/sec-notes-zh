## 什么是SQL注入？

SQL注入是一种攻击技术，攻击者通过在应用程序的输入字段中插入恶意的SQL代码，来操纵后端数据库的查询语句。其核心原理是**利用应用程序对用户输入数据验证或过滤不严的漏洞**。

一个典型的SQL查询语句如下：
```sql
SELECT * FROM users WHERE username = '[用户输入]' AND password = '[用户输入]';
```
如果应用程序直接将用户输入拼接到查询语句中，攻击者可以输入 `admin' --` 作为用户名。这将使查询变为：
```sql
SELECT * FROM users WHERE username = 'admin' --' AND password = '[用户输入]';
```
在SQL中，`--` 是注释符，这意味着其后的密码检查条件被忽略，攻击者可能无需密码就能以管理员身份登录。

## SQL注入的主要危害

以下是SQL注入攻击可能造成的主要后果：

*   **未经授权的数据访问**：攻击者可以读取数据库中的敏感数据，如用户信息、商业机密等。
*   **数据篡改**：攻击者可以修改、删除或插入数据库中的数据，破坏数据完整性。
*   **权限提升**：攻击者可能利用漏洞获得数据库甚至服务器的更高控制权限。
*   **拒绝服务**：恶意的SQL查询可能耗尽数据库资源，导致服务不可用。

## 如何预防SQL注入？

预防SQL注入的关键在于永远不要信任用户输入，并对输入进行严格的验证和处理。OWASP提供了详细的预防备忘单，核心措施包括：

*   **使用参数化查询（预编译语句）**：这是最有效的方法。它将SQL代码与数据参数分离，数据库会明确区分代码和传入的数据，从而防止数据被解释为代码。
    *   **不安全的方式（拼接字符串）**：
        ```python
        query = "SELECT * FROM users WHERE username = '" + username + "';"
        ```
    *   **安全的方式（参数化查询）**：
        ```python
        query = "SELECT * FROM users WHERE username = %s;"
        cursor.execute(query, (username,))
        ```

*   **对输入进行严格的验证**：使用白名单机制，只允许符合特定规则（如格式、类型、长度、范围）的输入通过。

*   **使用存储过程**：在数据库层定义并调用存储过程，同样可以隔离代码和数据。但需注意，存储过程内部若动态拼接SQL，仍可能存在注入风险。

*   **最小权限原则**：为数据库应用程序账户配置所需的最小权限，避免使用具有高级权限（如`root`、`sa`）的账户连接数据库。

*   **转义特殊字符**：在无法使用参数化查询时，对用户输入中的所有特殊字符进行转义。但此方法被视为次要防线，容易因遗漏而出错。

## 总结

本节课中我们一起学习了SQL注入漏洞。我们了解了SQL注入是通过将恶意SQL代码插入输入字段来攻击数据库的原理，认识到它可能导致数据泄露、篡改等严重后果。最重要的是，我们掌握了防御SQL注入的核心策略：首要且最有效的方法是**使用参数化查询**，并辅以输入验证、最小权限等安全实践。通过应用这些知识，你可以显著提升应用程序的数据库安全性。
