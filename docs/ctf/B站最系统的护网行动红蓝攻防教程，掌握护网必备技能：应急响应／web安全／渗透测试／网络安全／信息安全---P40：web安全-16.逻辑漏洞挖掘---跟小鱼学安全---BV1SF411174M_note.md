在本节课中，我们将要学习逻辑漏洞挖掘的第三部分，重点介绍三种在实战中非常常见且易于挖掘的逻辑漏洞类型：越权漏洞、支付逻辑漏洞和条件竞争漏洞。掌握这些漏洞的发现思路和利用方法，是进行有效安全测试的关键。

## 越权漏洞 🔓

![](img/fe68eb93c577b79ca4061e8b1985f079_12.png)

上一节我们介绍了逻辑漏洞的基本概念，本节中我们来看看第一种常见漏洞——越权漏洞。越权漏洞是指一个正常用户A可以执行本应只有用户B或更高权限用户才能执行的操作。它主要分为两类：横向越权（平行越权）和纵向越权（垂直越权）。

![](img/fe68eb93c577b79ca4061e8b1985f079_14.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_16.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_18.png)

横向越权指相同权限等级的用户之间可以互相操作对方的数据，例如用户A修改了用户B的个人信息。纵向越权指低权限用户（如普通用户）可以执行高权限用户（如管理员）才能执行的操作。

![](img/fe68eb93c577b79ca4061e8b1985f079_20.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_22.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_24.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_26.png)

漏洞产生的原因通常是对客户端提交的请求数据过分信任，而遗漏了服务端的权限校验。例如，程序仅通过Cookie判断用户身份，或通过可被篡改的`user_id`参数来判断，而未在服务端进行二次验证。

挖掘此类漏洞的核心思路是：**抓取数据包，分析并修改其中用于标识用户或权限的参数值**。

以下是挖掘越权漏洞的典型步骤：

1.  **识别敏感功能点**：关注所有涉及用户个人数据操作的功能，如查看/修改个人信息、订单、地址等。
2.  **拦截请求**：使用Burp Suite等工具拦截上述功能的HTTP请求。
3.  **分析参数**：仔细查看请求中的参数，寻找如`id`、`user_id`、`username`等可能标识用户身份或数据归属的参数。
4.  **修改并重放**：尝试修改这些参数的值（例如，将`id=1`改为`id=2`），然后重放请求，观察是否能访问或操作其他用户的数据。

![](img/fe68eb93c577b79ca4061e8b1985f079_28.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_29.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_31.png)

**核心操作**：`抓包 -> 修改参数（如 id, user_id） -> 重放请求`

![](img/fe68eb93c577b79ca4061e8b1985f079_33.png)

## 支付逻辑漏洞 💳

![](img/fe68eb93c577b79ca4061e8b1985f079_35.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_37.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_38.png)

在了解了如何挖掘越权漏洞后，我们来看看另一类高价值的漏洞——支付逻辑漏洞。这类漏洞的根源在于服务端未对客户端提交的支付相关参数进行充分校验，攻击者通过篡改这些参数，可能实现低价购买、零元购甚至“赚钱”的效果。

![](img/fe68eb93c577b79ca4061e8b1985f079_40.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_42.png)

支付流程中可能被篡改的参数包括：商品价格、购买数量、订单ID、支付状态、优惠券信息等。

![](img/fe68eb93c577b79ca4061e8b1985f079_44.png)

以下是几种常见的支付逻辑漏洞场景：

![](img/fe68eb93c577b79ca4061e8b1985f079_46.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_48.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_50.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_51.png)

*   **修改购买数量**：将正数数量改为负数（如 `quantity=1` 改为 `quantity=-1`），可能导致系统向用户账户加钱。
*   **修改商品价格**：在提交订单时，拦截请求并将商品价格修改为极低值（如 `price=100` 改为 `price=0.01`）。
*   **修改商品ID**：用低价商品的ID替换高价商品的ID，从而以低价购买高价商品。
*   **修改订单状态**：拦截支付结果回调的请求，将“支付失败”的状态修改为“支付成功”，从而欺骗系统完成订单。
*   **篡改优惠券/积分**：修改优惠券金额、使用次数限制，或无限刷取积分。
*   **测试数据包未删除**：开发者遗留的测试订单或商品（如0元商品），攻击者通过遍历ID或特定参数可能发现并利用。

![](img/fe68eb93c577b79ca4061e8b1985f079_53.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_54.png)

**核心思路**：在支付流程的每一个环节（下单、确认、付款）都尝试抓包，并修改与金额、数量、状态相关的参数。

## 条件竞争漏洞 ⏱️

![](img/fe68eb93c577b79ca4061e8b1985f079_56.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_58.png)

最后，我们来探讨一种需要“抓住时机”的漏洞——条件竞争漏洞。当多个线程或请求同时访问和操作同一段共享代码、变量或资源（如余额、优惠券库存、文件），而程序没有做好“锁”等并发控制时，就可能产生非预期的结果。

简单理解，就是**在极短时间内，同时向服务器发送大量相同的请求**，利用服务器处理请求的间隙，达到超出单次请求限制的效果。

![](img/fe68eb93c577b79ca4061e8b1985f079_60.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_62.png)

以下是条件竞争漏洞的常见场景：

![](img/fe68eb93c577b79ca4061e8b1985f079_64.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_66.png)

*   **领取优惠券/红包**：本来只能领取一次，通过并发工具同时发送数十个领取请求，可能成功领取多个。
*   **抽奖活动**：限制每日一次抽奖，利用条件竞争可能突破次数限制。
*   **余额充值/转账**：在余额校验和扣款之间存在的极短时间窗口内，发起并发操作，可能导致余额判断失效。

![](img/fe68eb93c577b79ca4061e8b1985f079_68.png)

一个特殊的案例是**条件竞争的文件上传**：有些上传逻辑是先将文件保存到服务器，再检查文件后缀是否合法，不合法则删除。攻击者可以在文件被保存但尚未被删除的极短时间内，疯狂并发访问该文件，有可能访问到“漏网之鱼”。

![](img/fe68eb93c577b79ca4061e8b1985f079_70.png)

**核心利用方法**：使用Burp Suite的Intruder模块，将可疑的请求（如领取、抽奖）设置为Payload，采用多线程、无间隔的爆破模式进行重放。

![](img/fe68eb93c577b79ca4061e8b1985f079_72.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_73.png)

## 总结 📝

![](img/fe68eb93c577b79ca4061e8b1985f079_75.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_77.png)

![](img/fe68eb93c577b79ca4061e8b1985f079_79.png)

本节课中我们一起学习了三种核心的逻辑漏洞：
1.  **越权漏洞**：通过抓包修改身份标识参数（如`id`），实现跨用户数据访问或操作。
2.  **支付逻辑漏洞**：通过抓包修改支付相关参数（如价格、数量、状态），实现不当得利。
3.  **条件竞争漏洞**：通过高并发发送请求，利用服务器处理请求的时序问题，突破次数或数量限制。

![](img/fe68eb93c577b79ca4061e8b1985f079_81.png)

挖掘逻辑漏洞的关键在于**清晰的业务理解**和**细致的参数分析**。你需要清楚每个功能点的正常业务流程，然后思考“如果这个参数被我控制会发生什么”。多练习、多思考，你会发现逻辑漏洞是Web安全测试中一片广阔的天地。
