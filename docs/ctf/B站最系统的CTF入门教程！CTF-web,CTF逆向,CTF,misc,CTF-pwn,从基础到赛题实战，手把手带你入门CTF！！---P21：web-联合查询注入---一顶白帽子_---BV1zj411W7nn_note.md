## 判断漏洞是否存在

![](img/90384b19f85cac5607d93c98b0793ea7_10.png)

SQL注入的核心是根据客户端返回的结果，判断提交的测试语句是否被数据库引擎成功执行。如果语句被执行，则说明存在漏洞。

![](img/90384b19f85cac5607d93c98b0793ea7_12.png)

![](img/90384b19f85cac5607d93c98b0793ea7_14.png)

以下是判断漏洞是否存在的方法：

![](img/90384b19f85cac5607d93c98b0793ea7_16.png)

*   一般使用英文的单引号 `‘` 或双引号 `“` 进行测试。
*   如果页面返回了SQL语句的语法错误，则很大可能存在注入漏洞。

![](img/90384b19f85cac5607d93c98b0793ea7_18.png)

**注意**：所有命令必须使用英文符号，使用中文符号可能导致难以排查的错误。

![](img/90384b19f85cac5607d93c98b0793ea7_20.png)

上一节我们介绍了漏洞存在的初步迹象，本节中我们来看看如何在靶场中具体操作。

在DVWA靶场的数据库注入板块，有一个输入用户ID的查询框。正常输入 `1`，会返回用户`admin`的信息。
当我们输入 `1‘`（数字1加一个单引号）时，页面返回了SQL语法错误。

![](img/90384b19f85cac5607d93c98b0793ea7_22.png)

![](img/90384b19f85cac5607d93c98b0793ea7_24.png)

这是因为后端查询语句可能类似：
```sql
SELECT ... FROM ... WHERE user_id = ‘$id‘
```
当我们输入`1‘`时，语句变为：
```sql
SELECT ... FROM ... WHERE user_id = ‘1‘‘
```
这导致单引号未正确闭合，从而引发语法错误。这个报错提示我们，此处可能存在SQL注入漏洞。

## 判断注入类型

![](img/90384b19f85cac5607d93c98b0793ea7_28.png)

发现漏洞后，需要判断注入点是数字型还是字符型，这决定了后续注入语句的构造方式。

![](img/90384b19f85cac5607d93c98b0793ea7_30.png)

![](img/90384b19f85cac5607d93c98b0793ea7_32.png)

判断方法如下：

![](img/90384b19f85cac5607d93c98b0793ea7_34.png)

*   输入 `1 and 1=1` 和 `1 and 1=2`，观察页面返回结果。
*   如果两次返回的结果**不同**，则通常是**数字型**注入。
*   如果两次返回的结果**相同**，则通常是**字符型**注入。

在靶场中测试：
输入 `1‘ and ‘1‘=‘1` 和 `1‘ and ‘1‘=‘2`，发现两者返回结果相同。
这是因为后端查询语句是字符型，类似：
```sql
SELECT ... FROM ... WHERE user_id = ‘$id‘
```
当我们输入`1‘ and ‘1‘=‘1`时，数据库为了进行比较，会进行隐式类型转换，将字符串`‘1‘=‘1‘`和`‘1‘=‘2‘`都转换为数字`1`，因此最终执行的查询条件与`id=1`无异，导致结果相同。

![](img/90384b19f85cac5607d93c98b0793ea7_36.png)

**核心区别**：
*   **数字型注入**：`and`后的逻辑判断会直接影响整个查询条件。
*   **字符型注入**：由于引号包裹和隐式转换，需要**闭合前面的引号**，并**注释掉后面的引号**，才能使`and`后的逻辑判断生效。

![](img/90384b19f85cac5607d93c98b0793ea7_38.png)

![](img/90384b19f85cac5607d93c98b0793ea7_40.png)

## 确定查询列数

![](img/90384b19f85cac5607d93c98b0793ea7_44.png)

![](img/90384b19f85cac5607d93c98b0793ea7_46.png)

在利用`UNION`进行联合查询前，必须知道前端查询语句返回的列数，因为`UNION`要求前后查询的列数必须一致。

![](img/90384b19f85cac5607d93c98b0793ea7_48.png)

![](img/90384b19f85cac5607d93c98b0793ea7_50.png)

我们使用 `ORDER BY` 子句来确定列数。

![](img/90384b19f85cac5607d93c98b0793ea7_52.png)

以下是操作步骤：

![](img/90384b19f85cac5607d93c98b0793ea7_54.png)

![](img/90384b19f85cac5607d93c98b0793ea7_55.png)

![](img/90384b19f85cac5607d93c98b0793ea7_57.png)

*   输入 `1‘ order by 1 -- ` （`--` 是注释符，用于注释掉原查询后面的语句）。
*   逐渐增加`order by`后的数字（如2, 3, 4...）。
*   当页面正常显示时，说明该列存在。
*   当页面报错（例如提示“Unknown column”）时，说明该列不存在，那么最后一个正常的数字就是查询的列数。

![](img/90384b19f85cac5607d93c98b0793ea7_59.png)

在靶场中测试：
输入 `1‘ order by 2 -- `，页面正常显示。
输入 `1‘ order by 3 -- `，页面报错。
由此可知，该查询语句返回 **2** 列数据。

**注意**：在URL中，注释符`--`后通常需要加一个空格，或者使用`#`（在URL中需编码为`%23`）。

## 确定数据回显位

知道列数后，我们需要确定这些列的数据在网页的哪个位置显示出来，即“回显位”。这通过`UNION SELECT`配合占位数字来实现。

`UNION`操作符用于合并两个或多个`SELECT`语句的结果集，它有两大核心要求：
1.  结果集中的列名总是等于第一个`SELECT`语句中的列名。
2.  每个`SELECT`语句必须有相同数量的列。

在靶场中，我们构造如下语句：
```
1‘ union select 1,2 -- 
```
这条语句的意思是：联合原始查询和我们自定义的查询`select 1,2`。如果页面正常显示，并且原本显示数据的地方出现了数字`1`和`2`，那么就说明这两个位置分别是第1列和第2列的回显位。

![](img/90384b19f85cac5607d93c98b0793ea7_61.png)

![](img/90384b19f85cac5607d93c98b0793ea7_63.png)

测试后，我们发现页面在“First name”处显示`1`，在“Surname”处显示`2`。这意味着，我们后续注入查询的第一列结果会显示在“First name”位置，第二列结果会显示在“Surname”位置。

![](img/90384b19f85cac5607d93c98b0793ea7_65.png)

**技巧**：有时网站只回显第一行数据。为了让`UNION`查询的结果显示出来，可以将原查询条件设为不成立（例如 `id=‘-1‘`），这样页面就会只显示我们`UNION`查询的结果。

![](img/90384b19f85cac5607d93c98b0793ea7_67.png)

## 利用联合查询获取数据

![](img/90384b19f85cac5607d93c98b0793ea7_72.png)

确定了回显位后，我们就可以替换`UNION SELECT`后的占位符，查询我们想要的数据了。

例如，我们想查询当前数据库的用户和密码（假设存在`users`表，内有`user`和`password`列）：
```
-1‘ union select user, password from users -- 
```
执行后，“First name”位置将显示用户名，“Surname”位置将显示密码哈希值，从而成功获取敏感信息。

## 总结

![](img/90384b19f85cac5607d93c98b0793ea7_74.png)

本节课我们一起学习了联合查询注入的完整流程：

![](img/90384b19f85cac5607d93c98b0793ea7_76.png)

![](img/90384b19f85cac5607d93c98b0793ea7_78.png)

1.  **判断漏洞**：通过添加单引号`‘`，观察是否引发SQL语法错误。
2.  **判断类型**：通过`and 1=1`和`and 1=2`测试，区分数字型与字符型注入。
3.  **确定列数**：使用`ORDER BY`子句递增测试，直到报错，从而确定查询结果的列数。
4.  **寻找回显位**：使用`UNION SELECT 1,2,3...`确定数据在页面中的显示位置。
5.  **获取数据**：将回显位数字替换为想要的查询语句（如`select user, password from users`），从而盗取数据库信息。

![](img/90384b19f85cac5607d93c98b0793ea7_80.png)

整个过程环环相扣，理解每一步的原理对于掌握SQL注入至关重要。请务必在合法的靶场环境中进行练习，巩固所学知识。
