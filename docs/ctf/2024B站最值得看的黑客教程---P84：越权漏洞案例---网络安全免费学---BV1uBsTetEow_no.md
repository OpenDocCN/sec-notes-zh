# 网络安全漏洞挖掘：P84：越权漏洞案例详解 🎯

在本节课中，我们将学习一种常见的网络安全漏洞——越权漏洞。我们将通过几个真实案例，详细拆解其原理、发现过程与利用方式，帮助初学者理解并识别此类安全问题。

![](img/93548ba242ac2fefd1a259d1f182c785_1.png)

## 概述

越权漏洞的核心在于，应用程序未能正确验证用户身份与所请求操作之间的权限关系，导致攻击者能够执行超出自身权限范围的操作。本节将重点分析几种典型的越权漏洞场景。

![](img/93548ba242ac2fefd1a259d1f182c785_3.png)

## 案例一：绑定信息越权

上一节我们介绍了越权漏洞的基本概念，本节中我们来看看一个典型的绑定信息越权案例。此类问题在各大网站和厂商中普遍存在。

首次登录账号后，系统常要求绑定邮箱或手机号。在绑定过程中，如果修改请求中绑定的账户ID参数，即可将自己的手机号等信息绑定到他人的账号上。

以下是漏洞的发现与验证步骤：

1.  **识别用户标识**：每个平台注册账号后，都会有一个唯一标识用户身份的ID。它可以是纯数字、英文单词或特殊字符，例如：`IDREG-7875-C06F-411`。
2.  **定位关键请求**：在绑定手机或邮箱的功能流程中，捕获网络请求数据包。
3.  **修改请求参数**：在请求数据包中，找到代表目标账户的ID参数（如 `user_id`），将其修改为其他用户的ID。
4.  **验证漏洞**：提交修改后的请求。如果绑定成功，则证明存在越权漏洞，攻击者可以将自己的联系方式绑定到任意用户账户。

此漏洞危害极大，曾有多家知名厂商因此遭受损失。例如，支付宝和HackerOne平台都曾出现类似的UID越权问题，并为此支付了高额漏洞赏金。

## 案例二：跳过验证步骤

接下来，我们看一个与上述漏洞原理相似的案例——跳过验证步骤。其根本原因同样是服务器未验证操作步骤间的逻辑绑定关系。

以某“珠宝贷”平台的密码重置功能为例：

1.  正常流程是：输入手机号 -> 获取验证码 -> 输入验证码 -> 进入设置新密码页面。
2.  漏洞在于：设置新密码的页面没有验证当前会话是否通过了前一步的“验证码校验”。

以下是测试流程：

1.  **梳理正常流程**：首先用自己的账号完整走一遍密码重置流程，记录下每个步骤的页面链接和所需参数。
2.  **定位关键接口**：重点关注“获取验证码”和“设置新密码”这两个环节的请求。
3.  **尝试越权操作**：在重置他人账户密码时，在获取验证码后，不进入验证码输入页面，而是直接访问“设置新密码”的页面链接。
4.  **输入新密码**：在该页面直接输入新密码并提交。如果密码重置成功，则说明存在漏洞，攻击者可以绕过验证码校验，直接重置任意用户密码。

这两个步骤（验证码校验和密码设置）在服务端被错误地处理为独立环节，缺乏必要的关联验证。

## 案例三：Token可预测

最后，我们来分析一个在邮箱验证码环节出现的“Token可预测”漏洞。这个案例需要一些简单的分析技巧。

假设在找回密码时，用户邮箱收到这样一个链接：
`http://www.target.com/find/emailpass.php?id=7804861&token=1458607665e10adc3949ba59abbe56e057f20f883e`

以下是分析挖掘流程：

1.  **分析链接结构**：
    *   `id=7804861`：很可能代表用户账户ID。
    *   `token=1458607665...`：一个32位由数字和小写字母组成的字符串，这**高度符合MD5加密的特征**（32位十六进制字符串）。

2.  **解密Token**：将Token值放入MD5在线解密网站尝试解密。如果解密出一串纯数字（例如 `1458607665`），则可推测该Token可能是由某个数字凭证经过MD5加密生成的。

![](img/93548ba242ac2fefd1a259d1f182c785_5.png)

3.  **收集样本，寻找规律**：
    *   为同一账号多次申请找回密码，获取多个Token。
    *   解密这些Token，观察得到的数字。你可能会发现它们非常接近，例如：`1458607665`, `1458607670`, `1458607678`... 只有末尾几位数在变化。
    *   为另一个不同账号申请找回密码，获取其Token并解密。你可能会得到类似 `1458611234` 的数字，与第一个账号的数字前缀不同，但结构相似。

4.  **推测与验证**：
    *   推测规则：Token = MD5( 固定前缀 + 可变后缀 )。其中“固定前缀”可能在一天内不变，或与用户ID弱相关；“可变后缀”范围很小（如3-4位数字）。
    *   验证漏洞：要重置用户A（ID=7804861）的密码，可以尝试枚举所有可能的“可变后缀”（例如0000-9999），结合推测的“固定前缀”生成Token进行爆破。一旦爆破成功，即可利用此Token和用户A的ID构造重置链接，从而接管其账户。

**核心问题**：用于身份验证的Token（凭证）其生成规则过于简单且可预测，导致攻击者能够伪造有效Token。

## 案例四：凭证发送至多账户

这个案例揭示了逻辑漏洞的另一种形式。在某些系统的密码找回或重要通知功能中，存在“同时向多个账户发送凭证”的问题。

漏洞场景：系统允许通过用户名、手机号、邮箱等多种方式找回账户。当输入一种信息（如手机号）时，如果该信息在系统内关联了多个账户，系统可能会错误地向所有关联账户发送相同的验证凭证或重置链接。

**利用方式**：攻击者只需知道目标用户的其中一个关联信息（如一个旧邮箱），即可触发凭证发送。如果该凭证能用于登录或重置密码，攻击者就可能接管目标用户的其他关联账户。

## 总结

本节课中我们一起学习了四种典型的越权与逻辑漏洞案例：
1.  **绑定信息越权**：通过修改请求中的用户ID参数，将个人信息绑定到他人账户。
2.  **跳过验证步骤**：业务流程步骤间缺乏绑定验证，允许攻击者绕过关键校验环节。
3.  **Token可预测**：身份验证凭证的生成算法存在缺陷，导致凭证可被猜测或枚举。
4.  **凭证发送至多账户**：系统错误地向同一标识信息关联的多个账户发送敏感凭证。

挖掘这类漏洞的关键在于**仔细梳理业务流程**，**对比分析正常与异常请求的差异**，并**大胆假设、小心验证**每一个参数的作用和关联性。对于初学者而言，从这些简单的逻辑漏洞入手，是理解网络安全攻防思维的绝佳起点。
