# 网络安全入门：P82：密码重置漏洞产生的原因 🔑

在本节课中，我们将学习密码重置功能中常见的两种安全漏洞：**任意密码重置漏洞**和**验证凭证回传漏洞**。理解这些漏洞的产生原理，是进行Web安全测试和漏洞挖掘的基础。

## 任意密码重置漏洞原理

上一节我们介绍了密码重置的基本流程，本节中我们来看看黑客如何利用流程缺陷进行攻击。

正常的密码重置流程如下：
1.  用户访问网站，提交密码修改请求。
2.  服务器接收请求并解析，然后传递给后端数据库。
3.  后端数据库执行密码修改操作，并将结果返回给服务器。
4.  服务器将解析后的结果返回给客户端。
5.  客户端得知密码修改成功，使用新密码登录。

然而，当攻击者（黑客）介入时，思路会发生改变。普通人只修改自己的密码，但黑客会尝试修改他人的密码。攻击者将自己的修改请求（目标为他人账户）发送到服务器。

关键在于，**服务器是否对请求者的身份进行了严格校验**。如果服务器没有识别请求者是否为账户的真正主人，就会直接将请求传递给数据库执行。数据库执行修改后，将成功结果经由服务器返回给攻击者。这样，攻击者就获得了目标账户的新密码，从而接管了该账户。

这种因服务器未校验用户身份而导致的漏洞，就是**任意密码重置漏洞**。攻击者的核心思路是突破“只能修改自己密码”的常规逻辑。如果尝试失败，则转向寻找其他漏洞点。

## 验证凭证回传漏洞原理

接下来，我们探讨另一种常见于邮箱/手机号找回密码场景的漏洞。

在通过邮箱找回密码的业务流程中，系统通常会向用户的邮箱发送包含**修改密码链接**或**验证码**的邮件。服务器通过链接中的特定参数或用户输入的验证码，来确认操作者是本人。

这个漏洞的发现依赖于我们已掌握的技能：使用工具（如Burp Suite）拦截和查看HTTP请求与响应数据包。

![](img/6b5924fe1eea58090dee4ce0df405f32_1.png)

以下是发现此漏洞的步骤：
1.  使用自己的手机号或邮箱，触发“找回密码”功能。
2.  系统会向你的联系方式发送验证码或链接。
3.  此时，拦截并查看“找回密码”请求的**返回数据包（Response）**。
4.  检查返回数据包中是否直接包含了发送给你的**验证码**或**重置链接**等凭证信息。

![](img/6b5924fe1eea58090dee4ce0df405f32_3.png)

如果返回包中携带了这些敏感凭证，就存在**验证凭证回传漏洞**。攻击者可以尝试将请求中的联系方式参数修改为受害者的，然后发送请求。如果服务器依然在返回包中回传了对应受害者的凭证信息，那么攻击者就能利用这些信息重置他人密码。

此漏洞产生的原因，正是**服务器错误地将敏感验证凭证放在了返回给客户端的响应数据中**。

**案例**：历史上DDCTF赛事中就曾出现过因验证码在响应包中回传而导致的任意密码重置漏洞。

---

本节课中我们一起学习了两种密码重置漏洞的原理：**任意密码重置漏洞**源于服务器对修改请求者身份校验的缺失；**验证凭证回传漏洞**则是因为服务器将验证码等敏感信息直接返回到了客户端响应中。理解这些原理是识别和防范此类安全风险的第一步。
