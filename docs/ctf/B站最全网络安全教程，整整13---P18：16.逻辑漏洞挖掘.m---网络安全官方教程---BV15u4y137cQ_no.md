## 🎯 越权漏洞

上一节我们介绍了逻辑漏洞的基本概念，本节中我们来看看最常见的越权漏洞。越权漏洞是指一个用户能够执行本应属于其他用户的操作。它主要分为两类：
*   **横向越权**：相同权限等级的用户之间，例如用户A可以操作用户B的数据。
*   **纵向越权**：不同权限等级的用户之间，例如普通用户可以执行管理员的操作。

漏洞产生的根本原因是服务器**过分信任客户端提交的数据**，而**遗漏了关键的身份权限校验**。

![](img/60d571e21a8a7452fc41f1cd106db68d_14.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_16.png)

### 漏洞挖掘思路

![](img/60d571e21a8a7452fc41f1cd106db68d_18.png)

挖掘越权漏洞的核心思路是：**抓取数据包，分析并修改其中用于标识用户或资源的参数**。

![](img/60d571e21a8a7452fc41f1cd106db68d_20.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_22.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_24.png)

以下是挖掘越权漏洞的关键步骤：

![](img/60d571e21a8a7452fc41f1cd106db68d_26.png)

1.  **识别关键参数**：在涉及用户个人数据的操作（如查看/修改资料、订单、地址）时，抓包观察请求中是否存在如 `userid`、`id`、`username` 等参数。
2.  **修改参数值**：尝试将这些参数值修改为其他用户的对应值（例如，将 `id=1` 改为 `id=2`）。
3.  **观察响应**：查看服务器返回的响应，判断是否成功访问或修改了其他用户的数据。

### 实战案例演示

以一个商城系统的“修改收货地址”功能为例：

1.  用户登录自己的账户，进入地址管理页面。
2.  点击修改某个地址，并抓取该请求包。
3.  在请求包中，发现参数 `address_id=9`（当前用户的地址ID）。
4.  将 `address_id` 的值修改为 `1` 或 `2` 等其他值并重放请求。
5.  如果页面成功显示或修改了其他用户的地址信息，则存在横向越权漏洞。

**核心操作**：`抓包 -> 寻找ID类参数 -> 修改ID值 -> 重放请求`

## 💰 支付逻辑漏洞

在理解了如何挖掘越权漏洞后，我们来看另一类“有利可图”的漏洞——支付逻辑漏洞。这类漏洞通常因为服务器未对支付流程的关键参数进行严格校验，导致攻击者可以篡改支付信息。

![](img/60d571e21a8a7452fc41f1cd106db68d_33.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_35.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_36.png)

其核心利用手法同样是：**抓包改包**。

![](img/60d571e21a8a7452fc41f1cd106db68d_38.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_40.png)

### 常见的攻击点

支付流程中可能存在漏洞的环节非常多，以下是一些典型的攻击点：

![](img/60d571e21a8a7452fc41f1cd106db68d_42.png)

*   **修改商品数量**：将购买数量改为负数（如 `quantity=-1`），可能导致系统向用户账户加钱。
*   **修改商品单价**：在订单确认或支付请求中，将商品价格改为极低值（如 `price=0.01`）。
*   **修改商品ID**：用低价商品的ID替换高价商品的ID。
*   **修改订单状态**：拦截支付结果回调，将“支付失败”的响应包替换为“支付成功”的响应包，从而欺骗系统。
*   **篡改优惠券/积分**：修改优惠券金额、使用次数限制或积分数量。
*   **利用未删除的测试数据**：通过遍历商品ID等方式，寻找开发遗留的测试商品（如0元购或超低价商品）。

### 实战案例简述

![](img/60d571e21a8a7452fc41f1cd106db68d_46.png)

案例：某CMS系统支付漏洞
1.  用户购买一个价值6000元的商品。
2.  在支付确认环节抓包，发现参数 `quantity=1`（购买数量）。
3.  将 `quantity` 的值修改为 `-1` 并重放请求。
4.  系统处理订单后，用户账户反而增加了6000元余额。

**核心思路**：对支付流程中任何客户端可控制的参数（数量、价格、ID、状态、优惠信息）都尝试进行篡改。

## ⚡ 条件竞争漏洞

![](img/60d571e21a8a7452fc41f1cd106db68d_54.png)

最后，我们来探讨一种需要“抓住时机”的漏洞——条件竞争漏洞。当多个线程同时访问和操作同一段代码或资源（如余额、优惠券库存），而程序没有进行合理的“锁”控制时，就可能产生非预期的结果。

![](img/60d571e21a8a7452fc41f1cd106db68d_56.png)

简单理解就是：**在极短时间内，同时向服务器发送大量相同的请求**，以绕过业务逻辑上的次数或数量限制。

### 典型应用场景

![](img/60d571e21a8a7452fc41f1cd106db68d_58.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_60.png)

以下是条件竞争漏洞常见的业务场景：

*   **领取优惠券/红包**：限制每人领取一次，但并发请求可能成功领取多次。
*   **抽奖活动**：限制每人一次机会，并发请求可能获得多次抽奖机会。
*   **余额充值/转账**：并发请求可能利用余额检查与扣款之间的时间差，实现“一分钱多用”。
*   **文件上传**：一种特殊场景，当服务器采用“先上传，后检查，不合格则删除”的策略时，在“检查”与“删除”的间隙，并发访问可能访问到本应被删除的恶意文件。

![](img/60d571e21a8a7452fc41f1cd106db68d_62.png)

### 利用方法

![](img/60d571e21a8a7452fc41f1cd106db68d_64.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_66.png)

利用条件竞争漏洞通常需要借助工具的并发功能：

![](img/60d571e21a8a7452fc41f1cd106db68d_68.png)

1.  抓取目标操作的数据包（如“领取优惠券”的请求）。
2.  将该请求发送到Burp Suite的Intruder模块。
3.  设置攻击模式为 **“Sniper”** 或 **“Turbo Intruder”**（用于高速并发），并设置大量线程同时发送该请求。
4.  观察结果，看是否成功突破了次数限制。

**核心操作**：`抓取限制性操作请求 -> 使用工具高并发重放 -> 验证是否突破限制`

## 📝 总结

![](img/60d571e21a8a7452fc41f1cd106db68d_73.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_75.png)

![](img/60d571e21a8a7452fc41f1cd106db68d_77.png)

本节课我们一起学习了三种核心的逻辑漏洞：

1.  **越权漏洞**：通过修改请求中的用户标识参数（如 `id`），非法访问或操作他人数据。关键在于**理解参数含义并大胆尝试修改**。
2.  **支付逻辑漏洞**：通过篡改支付流程中的关键参数（数量、价格、状态等），实现不当得利。关键在于**对支付全流程每个环节的请求进行测试**。
3.  **条件竞争漏洞**：通过高并发请求，利用服务器处理请求的时间差，绕过次数或数量限制。关键在于**使用工具进行并发测试**。

![](img/60d571e21a8a7452fc41f1cd106db68d_79.png)

逻辑漏洞的挖掘高度依赖对**业务逻辑的深入理解**和**灵活的测试思路**。它没有固定的模式，但核心方法论是相通的：**仔细分析每个客户端请求，思考每个参数的作用，并尝试以非正常的方式去操控它**。记住，扫描器难以发现这类漏洞，手动分析测试是唯一的途径。
