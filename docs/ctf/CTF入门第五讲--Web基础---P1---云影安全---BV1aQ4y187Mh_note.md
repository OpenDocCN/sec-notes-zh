## 热身：了解CTF Web题型

![](img/e9662a674668a115565f24ccfe29d946_33.png)

在开始深入学习之前，我们先通过几道简单的热身题来感受一下CTF Web题是什么样子的。这些题目通常来自BUUCTF或攻防世界等平台。

以下是几道典型的签到题：

![](img/e9662a674668a115565f24ccfe29d946_35.png)

![](img/e9662a674668a115565f24ccfe29d946_37.png)

![](img/e9662a674668a115565f24ccfe29d946_39.png)

1.  **Disable Button**：页面上有一个无法点击的按钮。通过浏览器开发者工具（F12）检查HTML结构，可以发现按钮有一个`disabled`属性。删除该属性后即可点击按钮获得flag。
    ```html
    <!-- 原始代码 -->
    <button disabled>Click me</button>
    <!-- 修改后 -->
    <button>Click me</button>
    ```
2.  **Cookie**：题目提示查看Cookie。在浏览器开发者工具的“网络”或“应用程序”标签页中查看Cookie，或者直接访问`/cookie.php`等路径，即可在响应中找到flag。
3.  **robots.txt**：`robots.txt`文件用于告知爬虫网站的访问规则。访问`/robots.txt`，可能会发现不允许爬虫访问的路径（如`/flag.php`），直接访问该路径即可获得flag。

这些题目虽然简单，但涵盖了Web安全中常见的信息获取点：前端代码、HTTP协议头和网站配置文件。

## 信息泄露

上一节我们通过热身题接触了信息泄露的简单形式。本节中，我们来看看信息泄露更广泛的含义和常见位置。

以下是几种常见的信息泄露源：

![](img/e9662a674668a115565f24ccfe29d946_70.png)

![](img/e9662a674668a115565f24ccfe29d946_72.png)

![](img/e9662a674668a115565f24ccfe29d946_76.png)

![](img/e9662a674668a115565f24ccfe29d946_78.png)

*   **注释**：开发者留在HTML或代码中的注释可能包含敏感信息，如后台路径、测试账号等。
    ```html
    <!-- 管理员后台：/admin.php -->
    ```
*   **robots.txt**：如前所述，此文件可能暴露不希望被公开访问的目录。
*   **备份文件**：网站备份文件（如`.zip`, `.bak`, `.tar.gz`, `.7z`）如果被遗留在Web目录下，可直接下载并查看源码。
*   **版本控制文件**：如`.git`、`.svn`目录。如果这些目录可被访问，攻击者可能利用工具（如`GitHack`）还原出完整的网站源代码。
*   **编辑器临时文件**：如Vim编辑器异常退出时产生的`.swp`文件。
*   **DS_Store文件**：macOS系统在目录中自动生成的`.DS_Store`文件，可能包含目录结构信息。
*   **HTTP头**：服务器响应的HTTP头有时会泄露服务器版本、框架信息等。

### 实战：通过HTTP头泄露获取Flag

![](img/e9662a674668a115565f24ccfe29d946_107.png)

![](img/e9662a674668a115565f24ccfe29d946_109.png)

![](img/e9662a674668a115565f24ccfe29d946_111.png)

![](img/e9662a674668a115565f24ccfe29d946_113.png)

我们来看一道综合性的题目。题目页面看似空白，需要按步骤挖掘信息。

![](img/e9662a674668a115565f24ccfe29d946_115.png)

![](img/e9662a674668a115565f24ccfe29d946_117.png)

![](img/e9662a674668a115565f24ccfe29d946_119.png)

![](img/e9662a674668a115565f24ccfe29d946_121.png)

1.  首先检查网页源码（F12），发现一个带有`onclick`事件的链接，但被`return false`阻止。我们可以直接构造请求访问其指向的路径（如`/secret.php`）。
2.  访问`/secret.php`后，页面提示需要从特定来源（Referer）访问，并使用特定的浏览器（User-Agent）。
3.  此时需要使用工具（如HackBar插件或Burp Suite）来修改HTTP请求。
    *   添加或修改 `Referer` 头为题目要求的网址。
    *   修改 `User-Agent` 头为题目要求的浏览器标识。
4.  再次发送请求，可能还会要求从本地（`127.0.0.1`）访问，此时需要添加 `X-Forwarded-For` 头为 `127.0.0.1`。
5.  正确构造请求后，即可在响应中获得flag。

![](img/e9662a674668a115565f24ccfe29d946_123.png)

![](img/e9662a674668a115565f24ccfe29d946_125.png)

**核心请求头修改示例**：
```http
GET /secret.php HTTP/1.1
Host: target.com
Referer: http://required-site.com
User-Agent: RequiredBrowser/1.0
X-Forwarded-For: 127.0.0.1
```

## 代码审计与PHP反序列化

![](img/e9662a674668a115565f24ccfe29d946_131.png)

![](img/e9662a674668a115565f24ccfe29d946_133.png)

在Web题目中，我们常常会拿到一部分服务器端的源代码（通常是PHP），这就需要我们进行代码审计，找出其中的逻辑漏洞。PHP反序列化是其中一类常见且重要的漏洞。

![](img/e9662a674668a115565f24ccfe29d946_135.png)

### 序列化与反序列化基础

![](img/e9662a674668a115565f24ccfe29d946_139.png)

![](img/e9662a674668a115565f24ccfe29d946_141.png)

![](img/e9662a674668a115565f24ccfe29d946_143.png)

*   **序列化**：将对象的状态信息转换为可以存储或传输的形式（字符串）的过程。
*   **反序列化**：将序列化后的字符串恢复为对象的过程。

在PHP中，使用 `serialize()` 和 `unserialize()` 函数进行这两个操作。当程序接收外部输入并对其进行反序列化时，如果输入可控，就可能触发漏洞。

### PHP魔术方法

![](img/e9662a674668a115565f24ccfe29d946_151.png)

![](img/e9662a674668a115565f24ccfe29d946_153.png)

![](img/e9662a674668a115565f24ccfe29d946_155.png)

![](img/e9662a674668a115565f24ccfe29d946_157.png)

![](img/e9662a674668a115565f24ccfe29d946_159.png)

PHP类中的一些特殊方法（魔术方法）会在特定时机自动调用，这与反序列化漏洞密切相关：

![](img/e9662a674668a115565f24ccfe29d946_161.png)

*   `__construct()`：对象创建时调用。
*   `__destruct()`：对象销毁时调用。
*   `__sleep()`：执行`serialize()`时调用。
*   `__wakeup()`：执行`unserialize()`时调用。

![](img/e9662a674668a115565f24ccfe29d946_163.png)

![](img/e9662a674668a115565f24ccfe29d946_165.png)

![](img/e9662a674668a115565f24ccfe29d946_167.png)

![](img/e9662a674668a115565f24ccfe29d946_169.png)

攻击者可以精心构造一个序列化字符串，当它被反序列化时，会触发对象中某些魔术方法的执行，从而达到执行任意代码或命令的目的。

### 实战：简单的反序列化题目

![](img/e9662a674668a115565f24ccfe29d946_183.png)

![](img/e9662a674668a115565f24ccfe29d946_185.png)

![](img/e9662a674668a115565f24ccfe29d946_196.png)

![](img/e9662a674668a115565f24ccfe29d946_198.png)

![](img/e9662a674668a115565f24ccfe29d946_200.png)

![](img/e9662a674668a115565f24ccfe29d946_202.png)

题目给出类似以下源码：
```php
<?php
class CTF {
    public $flag = 'flag{test}';
    function __wakeup() {
        exit('Bad request');
    }
}
if (isset($_GET['code'])) {
    unserialize($_GET['code']);
} else {
    highlight_file(__FILE__);
}
?>
```
**解题思路**：
1.  代码接收`code`参数并反序列化。
2.  反序列化时会触发`__wakeup()`方法，该方法直接退出脚本，阻止我们获取`$flag`属性。
3.  一个常见的绕过方法是利用PHP反序列化字符串的特性。当序列化字符串中表示对象属性数量的值大于实际属性数量时，`__wakeup()`方法可能不会被调用（在特定PHP版本下）。
4.  首先序列化一个`CTF`对象：`$a = new CTF(); echo serialize($a);` 得到 `O:3:"CTF":1:{s:4:"flag";s:9:"flag{test}";}`
5.  将属性数量`1`改为更大的数，如`2`：`O:3:"CTF":2:{s:4:"flag";s:9:"flag{test}";}`
6.  将修改后的字符串作为`code`参数传递，可能成功绕过`__wakeup()`，从而让程序正常执行到结束，有时就能泄露或输出`$flag`。

文件上传漏洞是指Web应用允许用户上传文件，但未对上传文件的类型、内容等进行严格检查，导致攻击者可以上传恶意文件（如Webshell），从而获取服务器控制权。

### 一句话木马

![](img/e9662a674668a115565f24ccfe29d946_230.png)

![](img/e9662a674668a115565f24ccfe29d946_232.png)

![](img/e9662a674668a115565f24ccfe29d946_234.png)

![](img/e9662a674668a115565f24ccfe29d946_236.png)

![](img/e9662a674668a115565f24ccfe29d946_238.png)

![](img/e9662a674668a115565f24ccfe29d946_242.png)

最常用的Webshell是“一句话木马”，其PHP版本如下：
```php
<?php @eval($_POST['cmd']); ?>
```
这段代码执行通过POST参数`cmd`传递的任何PHP代码。

### 常见绕过技巧

![](img/e9662a674668a115565f24ccfe29d946_252.png)

![](img/e9662a674668a115565f24ccfe29d946_254.png)

1.  **前端验证绕过**：仅通过JavaScript检查文件后缀。直接使用抓包工具（如Burp Suite）拦截上传请求，修改文件名后缀即可绕过。
2.  **Content-Type绕过**：服务器通过检查HTTP请求头的`Content-Type`字段来判断文件类型。抓包修改该值为合法的图片类型，如`image/jpeg`。
    ```http
    Content-Type: image/jpeg
    ```
3.  **文件头绕过**：服务器检查文件内容开头字节（魔术字节）。可以在木马内容前添加图片的文件头，如GIF的`GIF89a`。
    ```php
    GIF89a
    <?php @eval($_POST['cmd']); ?>
    ```
4.  **后缀名绕过**：尝试其他可执行后缀，如`php3`, `phtml`, `phps`等，或者利用系统特性，如上传`.htaccess`文件配置Apache服务器将特定后缀解析为PHP。
5.  **条件竞争**：服务器先允许文件上传，然后再进行安全检查并删除不安全文件。利用上传和删除之间的极短时间差，不断发起上传和访问请求，以期在文件被删除前访问到并执行它。

### 实战：基础文件上传

![](img/e9662a674668a115565f24ccfe29d946_265.png)

![](img/e9662a674668a115565f24ccfe29d946_267.png)

![](img/e9662a674668a115565f24ccfe29d946_268.png)

![](img/e9662a674668a115565f24ccfe29d946_270.png)

![](img/e9662a674668a115565f24ccfe29d946_272.png)

![](img/e9662a674668a115565f24ccfe29d946_274.png)

![](img/e9662a674668a115565f24ccfe29d946_275.png)

1.  找到一个文件上传点。
2.  尝试上传一个`.php`文件，通常会被拦截。
3.  使用Burp Suite抓取上传请求包。
4.  将文件名后缀改为`.jpg`，同时将`Content-Type`改为`image/jpeg`。
5.  如果服务器仅检查了这两项，文件可能上传成功。但此时`.jpg`文件不会被当作PHP执行。
6.  有时需要结合**文件包含漏洞**或**解析漏洞**。例如，如果服务器存在文件包含漏洞，可以尝试包含这个上传的图片马，其中的PHP代码仍会被执行。或者，某些旧版本服务器（如IIS 6.0）存在目录解析漏洞（`/upload/test.asp;.jpg` 会被当作`.asp`执行）。

Web安全涉及面广，技术迭代快。除了本节课介绍的内容，SQL注入、XSS（跨站脚本）、SSRF（服务器端请求伪造）、SSTI（服务器端模板注入）等都是重要考点。建议初学者在掌握基础后，多在BUUCTF、攻防世界等平台练习相关题目，并学会使用Burp Suite、HackBar等工具。记住，多见题型、多动手实践是提升CTF能力的关键。
