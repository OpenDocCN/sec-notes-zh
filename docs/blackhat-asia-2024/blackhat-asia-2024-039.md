# 039：攻击Android生态系统中的调试模块 🛡️

在本节课中，我们将学习Android生态系统中的调试模块，分析其安全风险，并通过实际案例了解攻击者如何利用这些模块中的漏洞。我们将从Android系统的碎片化背景讲起，逐步深入到厂商定制的调试架构、攻击面分析以及具体的漏洞案例。

## 概述

Android系统的复杂性源于其多层次的生态系统结构，这导致了严重的碎片化问题。为了进行故障排除、功能验证和工厂测试，Android及其硬件供应商引入了各种调试模块。然而，这些高权限的调试模块如果设计不当或未被妥善移除，就可能成为严重的安全隐患。本节课将系统性地剖析这些调试模块的架构、攻击面，并通过真实漏洞案例展示其潜在危害。

## 背景：Android生态系统的碎片化

上一节我们介绍了课程的整体目标，本节中我们来看看导致安全问题的根源——Android生态系统的碎片化。

Android生态系统由多个层级构成，如下图所示：

![](img/b2f824f963e5b7910dd51e83b2f8fc5a_1.png)

从底层开始：
1.  **系统芯片层**：包括高通、联发科、紫光展锐等主要芯片制造商，提供底层硬件和板级支持包能力。
2.  **Android开源项目层**：不仅提供SOC级别的能力，还为上游定制和开发提供开源代码。
3.  **产品层**：包括智能手机、平板电脑、垂直和AIoT设备。
4.  **OEM品牌层**：如三星、小米、华为和谷歌自研的Pixel设备。

这种分层结构导致了供应链的复杂性，并在Android生态系统内部引入了碎片化。

碎片化主要体现在三个层面：
*   **产品碎片化**：厂商定制自己的启动器和操作系统（如MIUI、Magic UI、HarmonyOS），并在系统级APK（包括调试模块、通知和设备互联功能）上进行客户化开发。
*   **系统碎片化**：厂商在AOSP的框架层进行定制化开发以适应其硬件特性。它们还在硬件抽象层实现底层接口，以桥接框架层和驱动层。
*   **BSP碎片化**：厂商定制与硬件相关的驱动模块，体现在视频处理、Wi-Fi、蓝牙、GPS、4G/5G、音频处理、显示加速和安全相关组件中。

基于以上分析，正是Android生态系统的碎片化导致了影响整个行业各个方面的无数安全问题。

为了应对碎片化带来的安全问题，谷歌实施了以下四项策略：
1.  **供应商分区**：在Android O中引入，专门用于存放供应商特定的BSP代码，以便更好地管理和隔离供应商特定组件。
2.  **硬件抽象层**：在用户层和驱动层组件之间建立桥梁，提供标准化接口和抽象层，使上层应用无需直接依赖特定驱动程序即可与底层硬件交互。
3.  **安全公告**：谷歌发布月度Android安全公告，包含由漏洞赏金计划报告的关键和高危漏洞，确保及时披露和了解各种供应商特定实现中存在的安全漏洞。
4.  **Android芯片组安全赏金计划**：2019年启动，旨在识别并奖励专门针对SOC的安全漏洞。遗憾的是，截至2023年5月，谷歌已终止ASRP。

这些策略反映了谷歌为增强Android生态系统安全性、降低碎片化相关风险所做的努力。

## Android原生调试架构

在了解了碎片化的背景后，我们来看看标准的Android调试架构是如何工作的。

Android的原生调试架构包含多个组件，如标准的Android架构图所示，从上到下包括：系统APK、Java API框架、原生C/C++库、Android运行时、硬件抽象层和Linux内核。

标准的Android调试架构主要服务于三个目的：
1.  **日志捕获**：包括捕获APP日志、内核日志和子系统日志（例如调制解调器、DSP、Wi-Fi、蓝牙）。这些日志用于分析和调试应用程序、系统组件和特定子系统的行为。
2.  **功能验证**：涉及验证摄像头、显示、硬件和GPU渲染等模块的功能。它确保这些组件按预期工作。
3.  **工厂测试**：指在制造和验证阶段对特定功能进行的测试和验证。通常涉及供应商特定的测试，以验证硬件和软件组件的正常运行。

谈到Android调试架构，首先想到的是开发者选项。开发者选项提供以下基本功能，这些功能也可以在Android标准开发文档中找到：
*   **常规选项**：包括内存管理、错误报告和OEM解锁等选项。
*   **调试相关选项**：包括USB调试和Android调试桥，允许在计算机和Android设备之间进行调试和通信。
*   **网络相关选项**：包括与Wi-Fi和蓝牙设置相关的选项。
*   **输入相关选项**：包括触摸反馈和输入设置。
*   **绘图相关选项**：包括显示布局边界等选项，可用于调试和优化用户界面。

以下是使用Android调试桥进行调试过程的解释：
如图所示，从左到右：
1.  用户在设置APK（特别是在Settings Provider中）启用USB调试，将标志`settings.global.adb_enable`设置为`1`。
2.  在框架层，有一个名为`android.debug.AdbManager`的系统服务持续监控标志`settings.global.adb_enable`。当检测到该标志被设为`true`时，它使用`ctl.start`方法启动原生守护进程`adbd`。
3.  此时，用户可以使用调试工具连接到`adbd`进行调试。

以下是Android中日志捕获的解释：
Android APP层的日志和原生层的日志都通过Java本地接口涉及，并由名为`liblog`的Android系统库访问。`liblog`通过Unix域套接字与原生守护进程`logd`保持持续通信，允许将应用层日志输出发送到`logd`。
当使用`logcat`捕获日志时，它也通过Unix域套接字与`logd`通信以读取相关的日志输出。通过`logd`的Linux规则，我们可以看到`logd`本身有权读取内核日志和系统文件。

基于对Android调试架构两个方面的分析，我们可以得出以下结论：
*   调试模块涉及复杂的IPC通信，包括供应商调用、Unix域套接字、内容提供程序、硬件接口定义语言等。
*   调试模块内部的数据流错综复杂。用户级数据被逐步传递到更高权限的原生守护进程和驱动程序。

## 厂商调试架构

介绍了Android的调试架构后，现在让我们将焦点转向厂商调试架构。首先，考虑一个问题：为什么厂商需要定制调试？

我总结了以下三点原因：
1.  **日志捕获**：需要捕获子系统的调试日志，并标准化调试能力，包括捕获所有模块（如MediaTek AEE和Unisoc的vialog）的调试信息。
2.  **功能验证**：涉及测试多个硬件相关功能，例如电话、连接性、硬件和定位。
3.  **工厂测试工具**：用于在工厂阶段对硬件组件（如屏幕等）执行基本检查。

让我们看一个真实案例：一加手机被发现存在一个后门漏洞，可能允许直接root设备。
该问题源于高通在手机组件中包含了一个定制的工程模式APK。攻击者可以利用工程模式系统APK中的漏洞获得设备的root访问权限。这凸显了以下两个问题：
1.  BSP通常包含工厂测试工具，这可能引入高风险的安全漏洞。如果这些工具在设备到达最终用户之前没有得到妥善保护或移除，攻击者可以利用它们获得对设备的未授权访问或控制。
2.  OEM或ODM通常缺乏安全意识，未能从出厂软件包中移除调试工具。这些疏忽可能使设备容易受到攻击，因为调试工具可能提供对设备功能的未授权访问或控制。

## 攻击面分析与模型

在深入了解了厂商调试架构及其风险后，本节我们将系统性地分析调试模块的攻击面。

下图说明了调试模块的攻击面分析，可以从四个部分进行划分：

1.  **系统APK视角**：它们容易受到来自网络的攻击和来自第三方APK的本地攻击。
    *   就网络攻击而言，系统APK可能开放套接字端口并接受异常参数以执行高权限操作。
    *   就本地攻击而言，系统APK通过导出的接口（如Activity、Service、Broadcast和ContentProvider）接受参数。由于系统APK具有高权限，如果被攻破，攻击者可以获得系统级访问权限并进行进一步的权限提升攻击。

2.  **框架层**：通过AIDL接收来自用户空间的参数。如果此层的调试组件中存在权限检查缺陷，则可能导致本地权限提升。

3.  **原生守护进程**：它们通常通过Unix域套接字从应用层接收数据。对数据的不当处理可能使它们容易受到攻击。

4.  **硬件抽象层服务和驱动程序**：HAL服务通过HIDL接收参数，驱动程序通过ioctl接收参数。如果这些参数没有得到妥善处理，这些模块也可能成为攻击目标。

以下是具体的攻击路径分类：

*   **攻击调试APP**：攻击路径涉及通过第三方APK调用APK的高权限组件。攻击目标是系统APK的导出组件，这可能导致本地权限提升或信息泄露。此外，系统APK可能监听套接字端口，这可能导致远程命令执行。
    下图展示了在真实案例中发现的两种类型的漏洞，我将在后续的案例研究部分详细解释这些漏洞。

*   **攻击调试守护进程**：攻击入口点是Unix域套接字。可能触发的安全问题包括内存破坏、信息泄露、命令注入等。

*   **攻击调试硬件抽象层服务**：攻击入口点是Unix域套接字或HIDL。可能触发的安全问题包括内存破坏、信息泄露和命令注入。

*   **攻击调试驱动程序**：攻击入口点是通过文件操作。可能触发的问题包括内存相关问题，如内存破坏和信息泄露。

通过对调试模块的漏洞研究，我们对三家主流厂商进行了调查，发现了数百个安全漏洞，获得了49个CVE信用分。

## 案例研究

基于前面的攻击面分析，本节我们将通过几个具体的漏洞案例，来直观地理解攻击是如何发生的。

**案例一：CVE-2022-20098 - 信息泄露漏洞**
受影响的调试模块是一个名为`AEEAED`的调试原生守护进程。该模块发布了一个名为`/dev/socket/com.mediatek.aee.aed64`的Unix域套接字。该模块接受来自UDS的参数。其中一个参数用于传输任意进程ID。AEE模块解析通过UDS传递的命令，提取PID并转储相应进程的内存信息，导致信息泄露。
右图显示了完整的概念验证，表明传输的PID可以是任意的。这允许恶意进程未经授权访问其他进程的内存信息，最终导致信息泄露。
另一个与信息泄露相关的问题是相应的调试APP。该APP记录了设备的各种唯一标识符，如IMEI、IMSI、呼叫地址等。

**案例二：CVE-2022-48382 - 内存破坏问题**
受影响的进程是一个名为`vendor.sprd.hardware.log@1.0-service`的调试服务。该进程的入口点是HIDL套接字，即左图中第一个图像中的Unix套接字。存在一个缓冲区漏洞。栈缓冲区`buf`的默认大小为4096字节。通过UDS从外部接收并传入函数的数据长度`len`可能超过4096字节，导致缓冲区溢出。
右图显示了一个POC，传递长度为10000字节的数据会导致受影响的调试HAL服务崩溃。

**案例三：CVE-2022-39118 - 驱动层内存破坏**
问题被识别为CVE-2022-39118，受影响的调试驱动程序是SPI驱动程序。攻击入口点是通过文件操作。这里的问题是一个越界写漏洞。当写入过长的字节序列时，会触发内核恐慌，POC如右图所示。

**案例四：CVE-2022-47339 - 本地权限提升漏洞**
受影响的进程是一个具有root权限的调试守护进程，名为`cmdservice`。攻击入口点是Unix套接字`/dev/socket/cmdsk`。默认情况下`cmdservice`未启用，仅当属性标志`sys.cmdservice.enable`被设置为启用时才会启动。`cmdservice`通过Unix套接字接受复杂参数，并最终将参数传递给`popen`执行命令。这最终导致命令注入，允许本地权限提升和以root权限执行任意命令。

**案例五：完整的漏洞利用链 - CVE-2022-27250**
这是一个重复出现的问题，最初由安全公司Sware发现。问题发生在一个名为`SPRDAutoSLT`的工厂测试工具中。此APP接受各种调试组件，如摄像头调试、电话调试、FM调试、蓝牙调试等。该APP监听端口7878并接受命令。命令列表包括执行shell命令的能力。但是，支持的命令可能因设备而异。通过连接到端口7878并发送任意shell命令，我们可以远程获得系统级权限。该APP的SELinux上下文是`sprd_autoslt_app`。

让我们进一步分析CVE-2022-47339。该漏洞允许以root权限执行任意命令，但有一些限制。首先，启用`cmdservice`标志需要系统级权限来设置属性。普通的shell或APK或第三方APK无法调用`cmdservice`。其次，`cmdservice`监听Unix域套接字`/dev/socket/cmdsk`，连接受到SELinux限制，并非每个第三方APK都能连接到`cmdsk`并发送命令。
我们检查了`cmdservice`的SELinux策略，发现`cmdsk`可以被SELinux上下文为`sprd_autoslt_app`的进程访问。
因此，攻击路径很清晰，如右图所示：
1.  攻击者发起远程攻击，在`SPRDAutoSLT`上执行命令，获得系统级权限并获取`sprd_autoslt_app`的上下文。
2.  然后，攻击者利用系统级权限执行`setprop`命令来启动`cmdservice`进程。
3.  最后，由于`SPRDAutoSLT` APP的SELinux安全上下文，攻击者可以直接向`cmdsk`发送命令，并以root权限执行任意命令。
最终结果是远程获取设备的root访问权限。值得注意的是，此root访问权限在`cmdservice`的SELinux安全上下文内，但影响是重大的。

## 总结与建议

本节课中，我们一起学习了Android调试模块的安全风险。最后，让我们总结讨论的要点并提出安全建议。

首先，调试模块覆盖系统的多个层级，从APK级别到驱动级别，导致存在多个攻击面，主要集中在进程间通信上。
其次，一些调试功能需要在进程间执行高权限命令，对这些命令的不当处理可能导致本地权限提升。
最后，工厂测试工具经常调用Wi-Fi、蓝牙和电话功能，对这些工具的不当处理可能导致信息泄露，例如暴露Wi-Fi地址、蓝牙地址、IMEI号码和其他敏感信息。

![](img/b2f824f963e5b7910dd51e83b2f8fc5a_1.png)

安全建议可以总结为以下三个方面：
1.  **对供应商而言**：某些调试模块不应发布给下游，例如工厂测试工具。
2.  **对OEM和ODM而言**：应基于特定需求有选择地选用BSP模块，而不是全盘接受。
3.  **对用户而言**：定期执行设备安全升级。

本节课中，我们系统性地分析了Android生态系统调试模块的架构、攻击面和安全风险，并通过多个真实漏洞案例加深了理解。希望这些知识能帮助开发者、安全研究人员和组织更好地保护Android设备的安全。
