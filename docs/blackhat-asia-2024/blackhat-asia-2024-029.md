## 跨缓存攻击简介 🎯

跨缓存攻击是一种流行的内核漏洞利用技术。它允许攻击者将一个缓存中的释放对象重新分配到另一个缓存中，从而将未知漏洞转化为已知漏洞，简化利用过程。

一个简单的UAF（释放后使用）示例如下：
1.  触发UAF，获取受害者对象A。
2.  将受害者对象所在的slab归还给页分配器。
3.  让目标缓存B重用受害者slab的页面。
4.  此时，受害者对象A被重新分配为对象B。
5.  对受害者对象A的操作将破坏对象B，进而利用被破坏的对象B获取root权限。

在更广泛的场景中，对象A和B也可以是页面或其他类型的内存区域。

![](img/9572d0af670075f296a01cad045bfc3d_5.png)

跨缓存攻击流行的原因有三点：
*   原始漏洞对象有时难以直接利用。
*   可以将未知漏洞转化为已知漏洞。
*   可以构建纯数据利用技术，以对抗日益增强的内核防护机制。

然而，跨缓存攻击因其步骤繁多、不确定性高而被称为不稳定技术。因此，研究如何提高其效率和稳定性至关重要。

## 内核内存管理基础 📚

在深入跨缓存攻击之前，我们需要了解一些内核内存管理的基础知识，特别是slab分配器和kmem缓存。

### Slab分配器视图
一个slab由多个页面组成，包含多个对象。每个slab的`objects per slab`（每个slab的对象数）和`order of pages`（页面阶数）信息可以通过`/proc/slabinfo`文件系统导出。

### Kmem缓存视图
kmem缓存为每个CPU设置了本地缓存（per-CPU slab）以加速分配。我们需要特别关注`partial list`（部分空闲列表），它可能包含许多slab，是跨缓存攻击中需要操作的关键部分。

### Partial List的操作与刷新
将slab放入partial list有一个确定性的方法：
1.  创建一个满的slab。
2.  将当前任务绑定到目标CPU（例如CPU 0）。
3.  释放该满slab上的任意一个对象，即可将该slab放入CPU 0的partial list中。此方法可用于增加partial list的长度。

partial list具有刷新特性。尽管它可以容纳多个slab，但其能持有的最大数量由`cpu_partial`参数限制，该参数同样通过`/proc/slabinfo`导出。

一旦partial list中的slab数量超过`cpu_partial`限制，就会触发刷新：
*   仍包含在用对象的slab会被移动到节点partial列表。
*   完全空闲的slab会被释放回页分配器。

## 跨缓存攻击的通用流程与挑战 ⚔️

上一节我们介绍了内存管理的基础，本节我们来看看跨缓存攻击的标准流程及其面临的核心挑战。

以下是跨缓存攻击的通用工作流程：
1.  将任务绑定到单个CPU（例如CPU 0）。
2.  执行碎片整理，清空所有部分空闲slab中的空闲对象。
3.  分配约 `objects_per_slab * cpu_partial + 1` 个对象备用。
4.  分配 `objects_per_slab - 1` 个对象作为前导对象。
5.  分配受害者对象。
6.  触发漏洞释放受害者对象。
7.  分配 `objects_per_slab + 1` 个对象作为后置对象。此步骤使受害者slab变为满slab。
8.  释放所有前导和后置对象。此步骤将受害者slab放入partial list，并确保其变为完全空闲。
9.  释放步骤3中分配的一个对象。当释放第`cpu_partial`个对象时，会触发partial list刷新，完全空闲的受害者slab将被回收到页分配器。
10. 大量喷射对象B以占用受害者slab的页面。至此，受害者对象A被重新分配为对象B。
11. 利用被破坏的对象B构建提权原语。

### 面临的挑战
跨缓存攻击主要面临两大挑战：

![](img/9572d0af670075f296a01cad045bfc3d_15.png)

![](img/9572d0af670075f296a01cad045bfc3d_16.png)

**挑战一：在受限的分配原语下如何丢弃受害者slab**
通常，我们需要触发partial list刷新来丢弃受害者slab。这要求我们能够分配大量对象并保持它们不被释放一段时间。然而，这两个条件有时很难满足：
*   随机kmem缓存、auto slab等缓解措施可能使我们难以找到合适的分配原语。
*   某些临时内核对象分配后很快被释放，无法长时间持有。

**挑战二：如何确定性地让高阶slab重用低阶slab**
如果是从大slab跨缓存到小slab，只需丢弃大slab并喷射足够多的小slab来重用释放的页面。但如果是从小slab跨缓存到大slab，简单地大量分配对象B是无效的。我们需要将小slab的低阶页面合并到大slab的高阶页面中，这需要精心的堆布局。

## 实战：优化跨缓存攻击 🛠️

上一节我们明确了挑战，本节我们将以一个具体的NPU漏洞为例，深入探讨如何解决这两大挑战，实现更高效的跨缓存攻击。

### 目标漏洞分析
该漏洞是一个由同事发现的NPU问题，影响Qcomm 4.14内核，可导致UAF或双重释放。利用此漏洞可以获得一个任意kmem缓存释放原语。为了使用此原语，我们必须精心准备受害者对象的内容，因此需要进行跨缓存攻击。

受害者对象是`NPU network command object`，从一个专用的kmem缓存中分配。通过`/proc/slabinfo`可以获取该缓存的基本信息：
*   slab阶数：0
*   对象大小：104字节
*   每个slab的对象数：39
*   `cpu_partial`：30

该专用缓存partial list中的slab不多，是一个不活跃的缓存，易于操作且干扰较少。

### 漏洞利用计划
以下是完整的利用计划：
1.  触发漏洞获取受害者对象。
2.  使用跨缓存攻击，让`file`对象占用被释放的受害者对象。
3.  利用任意释放原语触发一个UAF。
4.  使用30页表技术利用该UAF获取root权限。

整个利用是纯数据利用，但严重依赖跨缓存攻击，因此需要一个可靠的跨缓存攻击流程。

我们选择从`kmalloc-8k`缓存分配`file`对象。因此，这次跨缓存是从专用缓存（order 0）到`kmalloc-8k`缓存（order 3），正好包含了我们需要解决的两个挑战。

### 解决挑战一：受限原语下的slab丢弃
`NPU network command object`是一个临时对象，分配和释放都非常迅速。我们无法分配大量此类对象并长时间持有。

我们的解决方案是采用一种“冒险风格”的slab移动原语来丢弃受害者slab。

首先，了解一个有趣的`s lab move primitive`（slab移动原语）。它可以将一个CPU的slab移动到另一个CPU的partial list。例如，将CPU 1的slab移动到CPU 0：
1.  将任务绑定到CPU 1。
2.  通过分配`objects_per_slab`个对象，使CPU 1的本地slab变满。
3.  将任务切换到CPU 0。
4.  释放步骤2中分配的所有对象。此时，slab移动发生，该满slab从CPU 1移动到CPU 0的partial list。

通过重复此原语，我们可以向目标CPU的partial list放入可控数量的slab。这种方法可以替代通用流程中的步骤3和步骤9。

但对于本漏洞，我们一次只能分配一个对象并持有极短时间，因此需要更激进的方法——“冒险风格”slab移动。

设想一个场景：有N个任务（N > objects_per_slab）绑定到CPU 1，每个任务都通过调用NPU控制命令不断分配和释放对象。总会有某个时刻，至少有`objects_per_slab`个任务完成了分配但尚未释放。由于同时发生了`objects_per_slab`次分配，必然会在CPU 1上创建一个满slab。此时，如果我们将这些任务中的任何一个切换到CPU 0并释放对象，就会在CPU 0上触发释放操作，从而引发slab移动原语。最终，这个满slab将从CPU 1移动到CPU 0的partial list。

通过调整冒险任务的数量、被切换的任务数量以及冒险时机，我们可以相对稳定地向目标CPU的partial list移动一定数量的slab。这样，即使分配原语非常受限，我们也能添加足够的slab并成功丢弃空闲的受害者slab。

基于新的slab丢弃方法，我们为本漏洞优化了跨缓存攻击流程：
1.  使用冒险风格slab移动原语进行碎片整理并创建新slab。
2.  分配受害者对象。
3.  触发漏洞释放受害者对象。
4.  将受害者slab移动到CPU 1的partial list（暂不触发CPU 1的partial list刷新）。
5.  将受害者slab从CPU 1的partial list移动到CPU 0，触发CPU 0的partial list刷新。
6.  喷射`file`对象以占用受害者slab。

### 解决挑战二：确定性的高阶slab重用
现在我们来解决第二个挑战：如何让order 3的slab确定性地重用order 0的slab。这需要了解页分配器。

#### 页分配器基础知识
页分配器维护不同迁移类型的空闲区域。在空闲区域中，有不同阶的页面。特别是`PCP list`（每CPU页列表），用于加速order 0页面的分配和释放。

通常，内核空间通过`alloc_pages`从`unmovable`（不可移动）空闲区域分配页面，而用户空间通过`mmap`从`movable`（可移动）空闲区域分配页面。

![](img/9572d0af670075f296a01cad045bfc3d_18.png)

`PCP list`有几个特点：
1.  order 0页面的分配和释放总是优先使用它，其设计类似于栈。
2.  如果列表中的页面过多，会触发刷新，将页面添加到尾部。
3.  当低阶页面被回收到空闲区域时，页分配器倾向于将它们合并成高阶页面。

#### 确定性合并步骤
以下是实现确定性页面合并的步骤：
1.  将任务绑定到CPU 0。
2.  分配大量order 0页面（例如使用管道），为后续触发PCP list刷新做准备。
3.  从`unmovable`空闲区域分配几百个物理连续的order 0页面（同样使用管道）。
4.  通过每8个order 0页面释放1个，创建order 0页面空洞。此时CPU 0的PCP list中将存在许多空洞。
5.  执行优化跨缓存流程的第一步（碎片整理）。此时，新创建的slab（即受害者slab）将来自我们刚刚创建的页面空洞之一。
6.  占用所有其他order 0页面空洞（例如使用`ion`进行页面分配）。此时内存区域中不再有页面空洞。
7.  完成优化跨缓存流程的步骤2至5，丢弃受害者slab。受害者slab的页面被回收到页分配器。
8.  释放步骤3中分配的所有页面（除了已被`ion`占用的部分）。此时，该区域只剩下一个order 3页面，且被释放的受害者slab页面必然在其中。
9.  释放步骤2中创建的所有页面，强制触发PCP list刷新。此后，受害者slab页面和其他order 0页面被回收到空闲区域，页面合并将确定性地发生。受害者slab的order 0页面被合并到我们的order 3页面中。
10. 大量喷射`file`对象以占用这些order 3页面，完成跨缓存攻击。

#### 关键问题：分配物理连续的order 0页面
在实际操作中，整个利用的成功率很大程度上取决于步骤3：如何从`unmovable`空闲区域分配几百个物理连续的order 0页面。

我们通过侧信道技术探测页分配器状态来解决这个问题。观察发现，如果持续从`unmovable`区域分配带`__GFP_KSWAPD_RECLAIM`标志的order 0页面，页分配器会经历几个状态：
*   **状态1**：直接从PCP list分配。
*   **状态2**：PCP list变空，开始使用`unmovable`空闲区域。
*   **状态3**：`unmovable`区域变空，开始根据回退列表使用其他迁移类型的空闲区域。此时，如果空闲页数低于高水位线，会唤醒kswapd进行回收。
*   **状态4**：进入慢速路径，内核会采取多种操作回收页面。

![](img/9572d0af670075f296a01cad045bfc3d_20.png)

![](img/9572d0af670075f296a01cad045bfc3d_22.png)

幸运的是，这些回收操作涉及的LRU列表和shrinker列表中的页面数量信息，可以通过`/proc/vmstat`文件获取，并且普通应用可以读取。因此，通过观察这些值在持续分配页面时的变化，我们可以探测到页分配器何时进入状态3或状态4，即`unmovable`区域几乎为空。

![](img/9572d0af670075f296a01cad045bfc3d_24.png)

![](img/9572d0af670075f296a01cad045bfc3d_25.png)

基于此，我们制定分配策略：
1.  使用`ion`预留十几个高阶页面。
2.  使用上述技术创建并探测`unmovable`空闲区域的空状态。
3.  释放步骤1中预留的高阶页面。
4.  分配一些order 0页面以减少干扰。
5.  从`unmovable`区域分配几百个物理连续的order 0页面。在此条件下，order 0页面来自高阶页面的拆分，这保证了它们的物理连续性。

至此，第二个挑战也得以解决。

## 利用30页表提权与三星设备适配 🏆

成功完成第一阶段的跨缓存攻击后，我们将获得一个UAF。接下来使用30页表技术来利用它。

### 传统跨缓存与页表占用
这次我们使用传统方法来丢弃受害者slab，并通过喷射大量用户页表来占用被释放的slab。为了提高跨缓存成功率，我们在第一步使用前述方法使`unmovable`空闲区域几乎变空。

### 适配三星设备
三星设备具有物理KASLR（仅内核文本随机化）等缓解措施，这阻止了我们直接用30页表构建物理地址的任意读写原语。但我们仍然可以选择破坏内核对象来构建任意读写原语。

通过控制一个有效的页表项，使其指向一个由`ion`拥有的物理页，我们可以在用户空间通过用户虚拟地址访问该物理页。我们选择破坏`pipe buffer`来构建任意读写原语。

![](img/9572d0af670075f296a01cad045bfc3d_27.png)

![](img/9572d0af670075f296a01cad045bfc3d_28.png)

为此，我们需要让`pipe buffer`的页面紧跟在`ion`拥有的页面之后。这可以通过前面提到的分配物理连续order 0页面的技术来实现。然后，我们修改相应的PTE，从而完全控制`pipe buffer`，并轻松利用管道原语构建任意读写。

为了绕过三星设备的SELinux，我们选择攻击`security_compute_av`函数中使用的全局数据来计算AV（访问向量）。最终，我们成功实现了提权。
