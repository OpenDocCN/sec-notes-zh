## 存储型跨站脚本攻击

![](img/79084441785ab533bb09eab9eb7bbb6a_13.png)

![](img/79084441785ab533bb09eab9eb7bbb6a_15.png)

上一节我们介绍了SQL注入，本节中我们来看看另一种常见的Web攻击——跨站脚本攻击。存储型跨站脚本攻击是一种恶意攻击方式，攻击者将恶意脚本代码存储到目标网站的服务器上。当其他用户访问包含这些恶意代码的页面时，他们的浏览器就会执行这些脚本。

攻击流程如下：
*   **攻击者** 向一个允许用户提交内容的网站（如论坛、评论区）提交包含恶意脚本的数据。
*   **服务器** 在未进行充分过滤的情况下，将这些数据存储到数据库中。
*   **受害者** 访问一个会展示这些存储数据的页面。
*   **服务器** 从数据库取出数据并发送给受害者的浏览器。
*   **受害者的浏览器** 将数据作为页面内容的一部分进行解析，并执行其中的恶意脚本。

例如，攻击者可能在评论框中输入：
```html
<script>alert('你的Cookie是：' + document.cookie);</script>
```
如果网站没有过滤这个脚本，那么所有查看这条评论的用户都会弹出一个显示自己Cookie的窗口。攻击者可以利用这种方式窃取用户的会话Cookie，从而冒充用户身份。

![](img/79084441785ab533bb09eab9eb7bbb6a_17.png)

## 跨站请求伪造

理解了如何窃取用户身份后，我们来看看攻击者如何利用这个身份执行非授权操作。跨站请求伪造攻击诱使已登录目标网站的用户，在不知情的情况下执行非本意的操作。

攻击流程如下：
*   **用户** 已登录并认证了 `bank.com`，浏览器保存了会话Cookie。
*   **用户** 访问了攻击者控制的恶意网站 `evil.com`。
*   **恶意网站** 的页面中包含一个向 `bank.com` 发起转账请求的代码（例如一个隐藏的图片标签）。
*   **用户的浏览器** 在加载 `evil.com` 页面时，会自动向 `bank.com` 发送这个请求，并附上用户的会话Cookie。
*   **银行服务器** 收到带有有效Cookie的请求，认为这是用户的合法操作，从而执行了转账。

例如，恶意页面中可能包含这样一个图片标签：
```html
<img src="https://bank.com/transfer?to=attacker&amount=10000" width="0" height="0" />
```
当用户访问这个页面时，浏览器会尝试加载这个“图片”，实际上却是向银行发送了一个转账请求。

## 防御机制

面对上述攻击，开发者需要采取有效的防御措施。以下是几种关键的防御策略：

![](img/79084441785ab533bb09eab9eb7bbb6a_19.png)

![](img/79084441785ab533bb09eab9eb7bbb6a_21.png)

![](img/79084441785ab533bb09eab9eb7bbb6a_23.png)

### 1. 输入验证与输出编码
不要尝试自己编写复杂的过滤或净化函数，因为很容易出错。应该使用成熟框架提供的安全函数。核心原则是：**对用户输入进行严格的验证，并在将数据输出到页面时，根据上下文进行正确的编码**（如HTML编码、URL编码）。

![](img/79084441785ab533bb09eab9eb7bbb6a_25.png)

![](img/79084441785ab533bb09eab9eb7bbb6a_27.png)

### 2. 使用CSRF令牌
这是防御CSRF攻击最有效的方法之一。服务器在生成表单时，嵌入一个随机生成的、不可预测的令牌。当用户提交表单时，服务器会验证这个令牌是否匹配。由于恶意网站无法获取或预测这个令牌，因此无法构造出有效的请求。
```html
<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="随机生成的令牌值">
  <!-- 其他表单字段 -->
</form>
```

### 3. 同源策略与框架安全
浏览器实施**同源策略**，限制来自不同源（协议、域名、端口）的脚本进行交互。这为Web安全提供了基础保障。在使用`<iframe>`嵌入第三方内容时，可以利用`X-Frame-Options`或`Content-Security-Policy`响应头来防止页面被恶意网站嵌套，从而避免点击劫持攻击。

![](img/79084441785ab533bb09eab9eb7bbb6a_39.png)

![](img/79084441785ab533bb09eab9eb7bbb6a_41.png)

### 4. 安全的身份验证与会话管理
在实现OAuth等第三方登录时，要注意防范**会话固定攻击**。服务器在用户完成第三方认证后，应该立即创建一个新的会话ID，而不是继续使用认证过程中传递的旧会话ID。

本节课中我们一起学习了Web安全中两个至关重要的攻击向量：存储型XSS和CSRF。我们了解了它们的攻击原理、潜在危害以及关键的防御措施，包括输入验证/输出编码、CSRF令牌、同源策略和安全的会话管理。Web环境复杂多变，作为开发者，必须时刻保持安全意识，遵循安全最佳实践，并借助成熟框架的安全特性来构建应用，而不是尝试自己发明安全解决方案。
