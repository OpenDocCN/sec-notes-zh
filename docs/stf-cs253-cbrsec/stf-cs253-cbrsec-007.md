# 007：跨站脚本防御 🛡️

在本节课中，我们将要学习如何防御跨站脚本攻击。我们将回顾XSS的核心问题，探讨多种防御策略，包括输入转义、内容安全策略等，并理解纵深防御的重要性。

## 概述

上一节我们介绍了跨站脚本攻击的基本原理。本节中我们来看看如何防御这类攻击。核心问题是，当我们将不受信任的用户数据与代码结合时，如果用户输入经过特殊构造，可能导致攻击者的代码在用户浏览器中执行。我们的目标是转义用户输入，确保浏览器将其视为数据而非代码。

## 核心问题与防御思路

![](img/851e60d07db4bfd8ac2f987a4241f209_5.png)

跨站脚本攻击的本质是**HTML注入**。攻击者的代码在受害用户的浏览器中运行，可能窃取Cookie、会话令牌等敏感信息。

我们的解决方案是：在将用户输入与代码结合之前，对其进行**转义**或**净化**。关键在于，我们需要知道数据将被放置在哪个上下文中，因为不同的上下文需要转义不同的字符。

不受信任的用户数据可能来自：
*   HTTP请求（查询参数、表单字段、头部、Cookie）
*   数据库（数据可能由他人录入）
*   第三方服务（服务本身可能被入侵）

![](img/851e60d07db4bfd8ac2f987a4241f209_19.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_21.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_23.png)

关于转义时机，我们更倾向于在**渲染时**（输出时）进行转义，而非在存入数据库时。这是因为：
1.  在渲染时，我们确切知道数据将被放入哪个上下文（如HTML标签内、属性内、脚本字符串内），从而可以进行正确的转义。
2.  如果只在入库时转义，其他团队成员可能绕过规则直接向数据库写入未转义的数据。

## 使用框架内置的转义功能

大多数Web框架都内置了HTML转义功能，你应该使用这些功能，而不是自己编写。这遵循了“林纳斯定律”：在足够多的眼睛注视下，所有错误都是浅显的。广泛使用的、开源的框架代码更可能经过审查和测试。

然而，仅仅使用框架功能并不足够。你必须了解它具体在为什么上下文进行转义。例如，用于HTML转义的输出不能直接放入 `<script>` 标签内的字符串中，除非它也对JavaScript字符串的特定字符进行了转义。

以下是不同框架的例子：

![](img/851e60d07db4bfd8ac2f987a4241f209_25.png)

**EJS (Embedded JavaScript) 模板引擎**
EJS使用 `<%= value %>` 语法对值进行HTML转义并输出。
```javascript
// 模板
<h1>Hi <%= name %></h1>
// 渲染
ejs.render(template, { name: userInput }); // 自动转义
```
但需要注意，EJS中 `<%- value %>` 语法会输出**未转义**的HTML，这非常危险且容易误用。

![](img/851e60d07db4bfd8ac2f987a4241f209_27.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_29.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_31.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_33.png)

**React框架**
React默认会对通过插值 `{userInput}` 放入JSX的内容进行转义。
```jsx
<div>{userInput}</div> // 安全，会被转义
```
如果确实需要设置原始HTML，React提供了一个显式且“丑陋”的API `dangerouslySetInnerHTML`，以警示开发者。
```jsx
<div dangerouslySetInnerHTML={{__html: sanitizedHTML}} />
```
这种设计使得安全的行为是默认的，而“退出”安全行为则显得困难且显眼，有利于代码审查。

## 纵深防御策略

![](img/851e60d07db4bfd8ac2f987a4241f209_35.png)

纵深防御是指提供冗余的安全措施，即使一层防御失效，另一层也能提供保护。攻击者需要发现多个漏洞才能成功。

在XSS防御中，我们不能只依赖输入转义，还应假设攻击可能发生，并部署其他层级的防御。

![](img/851e60d07db4bfd8ac2f987a4241f209_37.png)

![](img/851e60d07db4bfd8ac2f987a4241f209_39.png)

### 防御层一：HttpOnly Cookie

即使攻击者能在页面中运行JavaScript，`HttpOnly` Cookie也无法通过 `document.cookie` 访问。这防止了通过XSS直接窃取会话Cookie。
```
Set-Cookie: sessionId=abc123; HttpOnly
```
Cookie仍会随HTTP请求自动发送到服务器，用于身份验证，只是对客户端JavaScript不可见。

### 防御层二：内容安全策略

内容安全策略是一个强大的、基于HTTP头部的防御机制。它允许网站管理员控制页面可以加载或执行哪些资源。即使攻击者成功注入脚本，CSP也可以限制其行为。

**CSP的核心思想**：通过白名单告诉浏览器，只允许执行或加载来自特定来源的脚本、样式、图片等资源。

一个简单的CSP策略：
```
Content-Security-Policy: default-src 'self'
```
这个策略表示：默认只允许加载来自**同源**的资源。

让我们分析一些例子，看看在上述策略下哪些内容会被允许或阻止：

![](img/851e60d07db4bfd8ac2f987a4241f209_41.png)

以下是CSP策略如何影响不同资源加载的示例：
*   `<script src="/local.js"></script>` **允许**（同源）
*   `<script src="http://evil.com/bad.js"></script>` **阻止**（不同源）
*   `<script>alert('xss')</script>` **阻止**（内联脚本默认被阻止）
*   `<div onmouseover="alert('xss')"></div>` **阻止**（内联事件处理器默认被阻止）

![](img/851e60d07db4bfd8ac2f987a4241f209_43.png)

默认阻止内联脚本和事件处理器是CSP防御XSS的关键，因为它迫使所有代码都放在外部文件中，而攻击者通常无法控制外部文件的来源。

**部署CSP的挑战**
在实际部署中，列出所有需要的资源来源可能很复杂，且容易遗漏导致网站功能损坏。可以采用“报告模式”逐步部署：
```
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-report
```
在此模式下，违规行为仅被报告而不会被阻止，方便管理员调整策略。

**CSP的演进与最佳实践：Strict Dynamic**
早期的CSP白名单模式存在缺陷，例如信任的域名上可能存在允许用户上传内容的端点，导致白名单被绕过。

现代的最佳实践是使用 `strict-dynamic` 配合随机数。其核心思想是：信任链传递。服务器生成一个随机数，并在HTTP头部和页面中合法的 `<script>` 标签上使用。
```
Content-Security-Policy: script-src 'nonce-abc123' 'strict-dynamic'
```
```html
<script nonce="abc123" src="https://trusted.cdn/library.js"></script>
<script nonce="abc123">
  // 合法的内联脚本
</script>
```
浏览器会执行带有正确随机数的脚本。由这些受信任脚本**动态创建**的其他脚本元素也会被允许执行。而攻击者注入的脚本没有正确的随机数，将被阻止。

对于不支持 `strict-dynamic` 的旧浏览器，可以同时提供回退方案，新浏览器会忽略旧指令。
```
script-src 'nonce-abc123' 'strict-dynamic' 'unsafe-inline' https:;
```

### 其他防御措施

*   **功能策略**：可以禁用某些浏览器API（如地理位置、摄像头），限制攻击者能做的事情。
*   **信任类型**：这是一个新的浏览器API提案，旨在解决DOM型XSS。它要求所有像 `innerHTML` 这样危险的DOM API只能接收特定的“受信任类型”对象，而这些对象必须通过预定义的、安全的净化函数创建。这迫使所有潜在的DOM XSS向量都通过一个集中的、可审计的安全函数。

## 总结

![](img/851e60d07db4bfd8ac2f987a4241f209_45.png)

本节课中我们一起学习了防御跨站脚本攻击的多层策略：
1.  **根本措施**：永远不要信任客户端数据，根据输出上下文进行正确的转义/净化，并优先使用框架的成熟功能。
2.  **纵深防御**：
    *   使用 **`HttpOnly` Cookie** 保护会话标识。
    *   部署强大的**内容安全策略**，并采用 `strict-dynamic` 与随机数的最佳实践，有效阻止未经授权的脚本执行。
    *   关注新的安全特性，如**信任类型**，以应对DOM型XSS。
3.  **保持警惕**：XSS是Web上最普遍的漏洞之一。安全是一个持续的过程，需要“恒常的警惕”。通过结合输入验证、输出转义、CSP等多层防御，可以极大地提升应用的安全性。
