## 回顾：API 设计缺陷

上一节我们讨论了 API 设计中的几种次优决策。本节中，我们来看看一个具体的、影响深远的例子：Node.js 中的 `Buffer` 类。

![](img/12d073e2d010137ccf484460158dc864_9.png)

![](img/12d073e2d010137ccf484460158dc864_11.png)

以下是几种常见的 API 设计问题：
*   **不安全的默认设置**：要求用户主动设置选项才能使 API 安全运行。
*   **令人困惑的函数签名**：根据传入参数的不同，函数执行完全不同的操作。
*   **基于参数数量的不同行为**：函数根据接收的参数数量改变其功能。

我们以 jQuery 和 Express 的 `use` 函数为例进行了说明。现在，让我们聚焦于 `Buffer` 类。

## `Buffer` 类的安全问题 🚨

`Buffer` 是 Node.js 中用于表示二进制数据的类型。它的构造函数 API 设计存在一个关键缺陷：根据传入参数的类型，行为截然不同。

**核心问题在于**：如果向 `new Buffer()` 传入一个数字（例如 `100`），它会分配指定字节数（100字节）的新内存，但**出于性能考虑，不会将其初始化为零**。这意味着返回的 `Buffer` 包含该内存地址上先前残留的任意数据（可能是敏感信息）。

![](img/12d073e2d010137ccf484460158dc864_13.png)

![](img/12d073e2d010137ccf484460158dc864_15.png)

```javascript
// 危险行为：分配未初始化的内存
let buf1 = new Buffer(100); // 可能包含进程内存中的敏感数据
let buf2 = new Buffer(100); // 每次内容都可能不同
```

![](img/12d073e2d010137ccf484460158dc864_17.png)

![](img/12d073e2d010137ccf484460158dc864_18.png)

![](img/12d073e2d010137ccf484460158dc864_20.png)

**安全的使用方式**是，在将 `Buffer` 返回给用户前，先将其填充为零：
```javascript
let buf = new Buffer(size);
buf.fill(0); // 安全：将内存清零
```

### 漏洞演示：一个简单的转换服务器

![](img/12d073e2d010137ccf484460158dc864_28.png)

![](img/12d073e2d010137ccf484460158dc864_29.png)

我们构建了一个服务器，它接收 JSON 数据，将其中的字符串转换为指定格式（如 Hex 或 Base64）。代码如下：

```javascript
app.post('/convert', (req, res) => {
    const data = JSON.parse(req.body);
    const str = data.str; // 用户控制此数据的类型
    const type = data.type;
    const result = convert(str, type); // 调用转换函数
    res.send(result);
});

function convert(str, encoding) {
    // 关键问题：`str` 可能不是字符串，而是数字
    return new Buffer(str).toString(encoding);
}
```

![](img/12d073e2d010137ccf484460158dc864_31.png)

![](img/12d073e2d010137ccf484460158dc864_33.png)

**漏洞成因**：用户通过 JSON 控制 `str` 字段的类型。如果用户传入 `{"str": 100, "type": "utf8"}`，`JSON.parse` 会将 `str` 解析为数字 `100`。当 `new Buffer(100)` 被执行时，它会分配 100 字节未初始化的内存，并将其内容作为 UTF-8 字符串返回，从而可能泄露进程内存。

**临时修复**：在转换前强制将输入转为字符串。
```javascript
function convert(str, encoding) {
    if (typeof str !== 'string') {
        str = String(str); // 强制类型转换
    }
    return new Buffer(str).toString(encoding);
}
```

### 真实世界的影响

这个设计缺陷在生态系统中广泛存在。例如：
1.  **`ws` 包**（流行的 WebSocket 库）：如果客户端发送的消息数据是数字，服务器会回送相应字节数的未初始化内存。
2.  **`request` 包**（流行的 HTTP 客户端）：如果附件数据是数字，会发送自身内存内容。
3.  **`bittorrent-dht` 包**：我们最初在此发现了该漏洞，攻击者可通过发送数字消息来 dump 对等节点的内存。

根本原因在于，`Buffer` 构造函数将两种完全不同的功能（**转换数据**和**分配内存**）合并到了同一个 API 中。

### 最终解决方案

Node.js 最终重构了 API，将功能分离：
*   `Buffer.from()`: 用于从各种数据**转换**创建 Buffer。
*   `Buffer.alloc()`: 用于**分配**指定大小且**已清零**的安全内存。
*   `Buffer.allocUnsafe()`: 用于分配**未初始化**的内存（高性能场景，开发者需明确知晓风险）。

同时，旧的 `new Buffer()` 构造函数被弃用。这是一个重要的教训：**默认行为应该是安全的**，危险操作需要显式选择。

### 另一个例子：`bcrypt` 库

![](img/12d073e2d010137ccf484460158dc864_41.png)

即使是安全敏感的库也可能犯类似错误。`bcrypt` 的哈希函数：
```javascript
bcrypt.hash(password, hashRounds, callback);
```
如果 `hashRounds` 意外被传入字符串（例如从环境变量读取，环境变量总是字符串），函数会将其误认为是盐值（salt），导致所有用户使用相同的盐，完全破坏了盐值的意义。

**教训**：应将不同功能拆分为不同的函数，避免基于类型的隐式行为。

## 服务器安全加固建议 🛡️

在深入本地服务器安全之前，我们先看几个通用的服务器安全最佳实践。

![](img/12d073e2d010137ccf484460158dc864_45.png)

### 隐藏详细错误信息

不应向用户暴露详细的堆栈跟踪，这可能会泄露文件路径、软件版本等敏感信息。

```javascript
// 在生产环境中隐藏堆栈跟踪
app.use((err, req, res, next) => {
    res.status(err.status || 500);
    if (process.env.NODE_ENV === 'production') {
        res.send('Internal Server Error');
    } else {
        res.send(`Error: ${err.message}\n${err.stack}`);
    }
});
```

## 核心议题：本地 HTTP 服务器安全 🖥️➡️🔓

![](img/12d073e2d010137ccf484460158dc864_80.png)

![](img/12d073e2d010137ccf484460158dc864_82.png)

现在，我们进入本节课的核心：运行在你计算机上的本地 HTTP 服务器所带来的安全风险。这些服务器可能是开发中的应用，也可能是已安装软件（如 Zoom）的后台服务。

### 风险原理

![](img/12d073e2d010137ccf484460158dc864_84.png)

![](img/12d073e2d010137ccf484460158dc864_86.png)

本地服务器通常监听 `localhost`（如 `127.0.0.1:4000`）。关键问题在于：**你浏览器中访问的任何网页，都可以向 `localhost` 上的端口发送请求**。

**简单演示**：一个打开字典应用的本地服务器。
```javascript
const exec = require('child_process').exec;
app.get('/', (req, res) => {
    exec('open /System/Applications/Dictionary.app', (err) => {
        if (err) return res.status(500).send(err.stack);
        res.send('Success');
    });
});
app.listen(4000, 'localhost');
```
访问 `http://localhost:4000` 会打开字典。但更重要的是，**任何你访问的网站**，只需在其页面中包含类似 `<img src="http://localhost:4000">` 的代码，就能触发这个请求，即使它无法读取响应。

### Zoom 漏洞案例分析

![](img/12d073e2d010137ccf484460158dc864_88.png)

![](img/12d073e2d010137ccf484460158dc864_90.png)

2020年，安全研究员发现 Zoom 客户端在 Mac 上运行了一个本地 HTTP 服务器，存在严重漏洞：
1.  **强制加入会议**：任何网站都能通过发送请求，在用户不知情的情况下将其加入 Zoom 会议并开启摄像头。
2.  **拒绝服务**：通过反复加入不存在的会议，使 Zoom 客户端不断抢夺焦点，导致电脑无法使用。
3.  **卸载残留**：即使用户卸载了 Zoom，这个本地服务器仍会残留，并可被用来重新安装 Zoom。

![](img/12d073e2d010137ccf484460158dc864_92.png)

**漏洞流程**：
1.  用户访问恶意网站。
2.  网站返回的页面包含向 `http://localhost:19421/launch?action=join&confno=ATTACKER_MEETING_ID` 发送请求的代码。
3.  请求到达本地 Zoom 服务器，触发加入会议操作。
4.  攻击者作为会议主持人，可设置“参会者加入时自动开启视频”规则。

![](img/12d073e2d010137ccf484460158dc864_94.png)

**根本原因**：本地服务器接受来自任何来源的 GET 请求，且功能强大（可执行加入会议等操作），同时缺乏必要的授权验证。

### 如何防御？探讨解决方案

假设我们必须使用本地服务器，如何保护它？

**思路一：检查 `Origin` 头**
服务器可以检查 HTTP 请求中的 `Origin` 头，只允许来自可信域名（如 `zoom.us`）的请求。
```javascript
app.use((req, res, next) => {
    const origin = req.get('Origin');
    if (origin && origin === 'https://zoom.us') {
        next(); // 允许请求
    } else {
        res.status(403).end(); // 拒绝请求
    }
});
```
**问题**：对于“简单请求”（如 `<img>`、`<script>` 标签触发的 GET 请求），浏览器**不会**发送 `Origin` 头。因此攻击者仍可通过嵌入图片等方式发起请求。

**思路二：使用“预检请求”**
对于“非简单请求”（例如使用 `PUT`、`DELETE` 方法，或设置了自定义头的请求），浏览器会先发送一个 `OPTIONS` 方法的**预检请求**来询问服务器是否允许该跨源请求。

**防御流程**：
1.  将本地服务器的 API 设计为需要 `PUT` 等非简单方法。
2.  浏览器在发送实际 `PUT` 请求前，会先发 `OPTIONS` 预检请求。
3.  服务器在 `OPTIONS` 处理中，检查 `Origin` 头，并返回允许的域和方法。
4.  浏览器确认后，才发送真正的 `PUT` 请求。
5.  服务器处理 `PUT` 请求。

![](img/12d073e2d010137ccf484460158dc864_96.png)

![](img/12d073e2d010137ccf484460158dc864_98.png)

这样，即使恶意网站发起请求，也会在预检阶段被拒绝，因为其 `Origin` 不在白名单内。

**然而**：这种方法仍然无法防御来自本机其他原生应用（如终端中的 `curl`）的请求，因为它们可以伪造任何头。更重要的是，存在一种名为 **DNS 重绑定** 的攻击，可以绕过同源策略，让恶意网站“变成” `localhost` 的来源。我们将在下节课详细讲解这种攻击。

## 总结 📚

本节课我们一起学习了：
1.  **API 设计安全**：以 `Buffer` 为例，说明了不安全的默认值、基于类型的多态行为所带来的严重安全隐患。安全的设计应使默认行为安全，并将不同功能分离。
2.  **服务器安全加固**：包括隐藏错误详情、移除服务器标识、限制监听地址等基本实践。
3.  **本地 HTTP 服务器安全**：深入分析了本地服务器（以 Zoom 为例）的巨大风险。任何网站都可能与你计算机上的本地服务交互，如果该服务存在漏洞，后果严重。我们探讨了通过检查 `Origin` 头和利用预检请求进行防御的思路，但也指出了其局限性。

![](img/12d073e2d010137ccf484460158dc864_100.png)

关键要点：**尽量避免创建本地 HTTP 服务器**。如果必须创建，应极度谨慎，假设它将受到整个互联网的攻击，并实施严格的身份验证和授权机制。本地服务的安全边界非常模糊，是攻击者的重要目标。
