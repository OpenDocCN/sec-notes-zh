# 012：现实世界中的HTTPS

在本节课中，我们将学习HTTPS在现实世界中的应用与挑战。我们将探讨两种重要的安全机制：HTTP严格传输安全（HSTS）和公钥固定（Key Pinning），了解它们的设计初衷、实际部署中遇到的问题以及未来的开放性问题。

## HSTS：解决HTTP降级攻击 🔒

上一节我们介绍了HTTPS的基本原理。本节中我们来看看一个现实问题：即使服务器支持HTTPS，用户的首次请求或页面中的某些链接仍可能通过不安全的HTTP协议发出，这为中间人攻击提供了可乘之机。HSTS正是为了解决这个问题而设计的。

HSTS允许网站通过一个HTTP响应头，告知浏览器在未来一段时间内，所有与该站点的通信都必须使用HTTPS。

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

以下是HSTS头中几个关键指令的含义：
*   **max-age**：指定HSTS信息在浏览器中缓存的时间（秒）。
*   **includeSubDomains**：此策略适用于该站点的所有子域名。
*   **preload**：一个用于加入浏览器内置HSTS预加载列表的指令。

### HSTS的优势与“锐利的边缘”

HSTS能有效防止协议降级和证书错误绕过，但它也存在一些设计上的挑战。

以下是HSTS部署中可能遇到的主要问题：
1.  **首次访问问题**：用户首次访问某个网站时，浏览器尚未收到HSTS头，因此该次请求可能被拦截。
2.  **包含子域名的陷阱**：启用`includeSubDomains`后，策略将覆盖所有子域名。对于拥有众多子域名的大型组织，若某个子域名不支持HTTPS，则会阻碍整个域的HSTS部署。
3.  **状态持久且不可逆**：一旦HSTS头被浏览器接收并缓存，在有效期内无法撤销。这要求网站在部署时需谨慎，通常从较短的`max-age`开始逐步增加。
4.  **HSTS追踪（超级Cookie）**：由于HSTS状态可以被第三方上下文（如嵌入的广告脚本）设置和读取，且不易被用户清除，它可能被滥用于跨站追踪用户。

### 应对HSTS追踪的尝试

针对HSTS追踪问题，浏览器厂商提出了不同的缓解方案。Safari的策略是限制第三方上下文设置和读取HSTS状态。Chrome则考虑利用其逐步淘汰“混合内容”（HTTPS页面加载HTTP子资源）的计划，从根本上消除通过观察HTTP请求是否被重写来读取“HSTS Cookie”的可能性。

## 公钥固定：强化证书认证 🔑

上一节我们讨论了如何强制使用HTTPS。本节中我们来看看如何确保连接到的HTTPS服务器是真实的。传统的证书体系允许任何受信任的证书颁发机构（CA）为任何域名签发证书。如果CA被入侵，攻击者就能签发虚假证书来冒充合法网站。公钥固定旨在解决这一问题。

公钥固定的核心思想是：网站运营商告知浏览器：“只接受使用特定公钥（或来自特定CA）的证书来认证我的网站。”这样，即使其他CA被入侵签发了虚假证书，浏览器也会拒绝连接。

网站可以通过HTTP响应头来声明其公钥指纹：

```
Public-Key-Pins: pin-sha256="base64=="; max-age=5184000
```

### 公钥固定为何失败？

尽管公钥固定在理论上很强大，但在实际部署中遇到了巨大挑战，最终导致Chrome移除了对动态公钥固定的支持。

以下是公钥固定面临的主要难题：
1.  **部署复杂度极高**：网站运营者需要深入理解公钥基础设施（PKI）、证书链等概念，并正确提取和配置公钥指纹，操作极易出错。
2.  **证书链构建的不确定性**：浏览器在验证证书时，可能会构建出与服务器发送的不同的证书链（例如，使用不同的根证书或遇到交叉签名）。这使得网站运营商几乎无法预测所有客户端会看到哪种链，从而难以正确设置固定策略。
3.  **失败的后果严重**：如果固定验证失败，浏览器会完全阻止访问，且用户无法“点击通过”错误页面。这种不可恢复、不可操作的体验对用户极不友好。
4.  **存在“敌对固定”风险**：如果攻击者入侵了服务器，他们可以设置恶意的固定策略，将用户锁定到攻击者控制的密钥上，甚至在攻击结束后仍导致用户无法访问恢复后的合法网站。

### 公钥固定的替代方案

由于动态公钥固定的种种问题，业界转向了其他补充方案：
*   **CAA记录**：通过DNS记录指定允许哪些CA为你的域名签发证书。这更多是一种事前预防和事后追责的机制，依赖于CA的自觉遵守。
*   **静态固定列表**：浏览器（如Chrome）内置一个经过严格审核的高价值网站列表及其公钥。这种方式更可靠，但扩展性差，网站难以加入，且密钥轮换不灵活。
*   **证书透明度**：这是一个旨在让所有证书签发行为公开可查的生态系统。我们将在下一节详细讨论。

## 证书透明度：让证书签发可见 📜

上一节我们提到公钥固定试图阻止恶意证书的使用。本节我们看看另一种思路：如果不一定能阻止，能否至少让所有签发的证书（包括恶意的）都公开可见，从而易于被发现和追责？这就是证书透明度的目标。

证书透明度要求CA将签发的每张证书提交到公开的、仅可追加的日志服务器中。任何观察者（如网站运营商、安全研究人员）都可以监控这些日志，查找未经授权签发的证书。

### CT如何工作？

其核心挑战在于如何确保日志本身是可信的。CT通过一个巧妙的密码学结构——默克尔树哈希——来实现。
*   日志会定期发布一个简短的“树哈希”作为状态摘要。
*   当CA提交证书时，日志会返回一个“承诺”，证明该证书已被接收。
*   服务器在TLS握手时，除了提供证书，还会提供来自一个或多个日志的“承诺”。
*   浏览器可以验证这些“承诺”的有效性。观察者之间可以通过比较“树哈希”来确保他们看到的是日志的完整且一致的状态。

### CT的价值与局限

证书透明度带来了显著好处：
*   **提高攻击成本与可检测性**：攻击者无法再悄无声息地使用恶意证书。
*   **发现错误与不良行为**：许多非恶意的证书配置错误、CA违规操作都是通过CT被发现的。

然而，CT也有其局限：
*   **重在检测，而非预防**：从恶意证书被签发到在日志中被观察到，存在一个时间窗口，攻击在此期间可能发生。
*   **系统复杂性高**：构建一个真正去中心化、能有效防范日志作恶的完整生态系统，仍有许多开放性问题待解决。

## 总结与开放性问题 🚀

本节课中我们一起学习了HTTPS在现实世界中面临的挑战以及相应的增强机制。

我们探讨了HSTS如何强制使用HTTPS并分析了其隐私追踪风险；深入了解了公钥固定的理想与它在部署复杂性、用户体验和灵活性方面的现实冲突；最后介绍了证书透明度作为一种补充性的审计和检测方案。

正如课程所展示的，在亿级用户的开放网络生态中推进安全变革是极其复杂的。我们面临着诸多尚无完美解决方案的开放性问题：
1.  **HSTS追踪**：如何在保障安全的同时，妥善处理其带来的隐私风险？
2.  **静态列表的负担**：如何替代或优化日益庞大的浏览器内置安全列表（如HSTS预加载列表）？
3.  **公钥固定的继承者**：是否存在一种方案，能提供类似公钥固定的强认证保证，同时避免其可用性陷阱？
4.  **证书透明度的完善**：如何构建一个真正健壮、无需信任单个日志的CT生态系统？

![](img/46709428f61e1956b0b0048c5dc4c803_2.png)

解决这些问题需要持续的研究、广泛的协作以及对安全、隐私和可用性之间微妙平衡的深刻理解。这正是网络安全领域令人兴奋的所在。
