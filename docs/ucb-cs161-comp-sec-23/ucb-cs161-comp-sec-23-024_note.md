# 024：入侵检测 🛡️

![](img/b9b362cdf38988dc91fd551839ebf112_1.png)

在本节课中，我们将要学习入侵检测。到目前为止，本课程主要关注如何预防攻击。然而，防御措施并非总是完美无缺。本节课将探讨当攻击发生时，如何检测它们。我们将从路径遍历攻击入手，作为检测的动机示例，然后深入探讨入侵检测系统的不同类型、如何衡量其有效性，以及检测攻击的各种策略。

## 路径遍历攻击 🗂️

上一节我们介绍了拒绝服务攻击和防火墙。本节中，我们来看看一种新的攻击方式，作为我们讨论入侵检测的起点。

路径遍历攻击利用了文件路径中的特殊字符。在终端中，`.` 代表当前目录，`..` 代表父目录。攻击者可以通过在用户输入中注入 `../` 等序列，使后端程序访问预期目录之外的文件。

例如，一个网站可能允许用户请求图片。后端代码可能将用户输入拼接到一个基础路径（如 `/home/public/`）之后。如果用户输入 `../private/passwords.txt`，最终形成的路径 `/home/public/../private/passwords.txt` 将指向 `/private/passwords.txt` 文件，从而让攻击者访问到未授权的敏感文件。

本节课的重点不是如何预防此类攻击（例如，通过过滤或转义特殊字符），而是假设防御措施可能失效，我们如何检测到攻击已经发生。

## 入侵检测系统的部署位置 📍

现在，我们知道了要检测什么。接下来需要决定将检测器放在网络的哪个位置。主要有三种选择。

### 网络入侵检测系统 🌐

第一种选择是将检测器部署在网络边界，例如路由器处。这种系统称为网络入侵检测系统。它的作用是监控所有进出网络的流量数据包，分析其中是否包含攻击迹象。

以下是 NIDS 的优缺点：

*   **优点**：成本低廉，一个系统即可保护整个内部网络；易于扩展和管理；对终端用户透明；信任计算基较小。
*   **缺点**：检测器对数据包的解读可能与最终接收数据包的主机不一致。例如，数据包的TTL（生存时间）可能使其无法到达目标主机，但NIDS仍会报警。此外，攻击者可能使用URL编码等方式绕过基于简单字符串匹配的检测。最棘手的问题是，如果流量经过TLS等加密，NIDS将无法解读数据包内容，从而难以检测攻击。

### 基于主机的入侵检测系统 💻

第二种选择是在网络中的每台主机（或服务器）上单独安装检测器。这种系统称为基于主机的入侵检测系统。

以下是 HIDS 的优缺点：

*   **优点**：解决了NIDS的解读不一致问题，因为检测器就在最终处理数据的主机上；能够解读加密流量；可以检测到网络内部的攻击。
*   **缺点**：需要在每台主机上部署和管理，成本高昂，扩展性差；可能无法覆盖所有老旧或未知的系统。

### 日志分析 📝

第三种方法不部署额外的检测器，而是利用系统或服务自身生成的日志文件。通过编写脚本（例如Python脚本）定期分析这些日志，寻找攻击证据。

以下是日志分析的优缺点：

*   **优点**：成本极低，无需构建专用检测系统；通常不存在解读歧义。
*   **缺点**：只能进行事后检测，无法实时阻止攻击；如果攻击者能够篡改或删除日志文件，则会失去检测依据。

## 如何衡量检测器的好坏 📊

我们已经了解了在哪里部署检测器。接下来，我们需要知道如何评估一个检测器的性能。检测器在回答“是否存在攻击”时，可能犯两种错误。

衡量检测器性能需要关注两个指标：
*   **误报率**：在**没有攻击**发生的情况下，检测器错误地发出警报的概率。公式可表示为：`误报率 = 错误报警次数 / 无攻击事件总数`。
*   **漏报率**：在**确实发生攻击**的情况下，检测器未能发出警报的概率。公式可表示为：`漏报率 = 未检测到的攻击次数 / 攻击事件总数`。

理想的检测器应同时降低这两种错误率，但现实中往往需要在两者之间进行权衡。例如，一个永远不报警的检测器漏报率为100%，误报率为0%；而一个永远报警的检测器漏报率为0%，误报率为100%。这两种都没有实际用处。

评估时还需考虑具体场景的成本。例如，对于火灾警报，误报的成本是打扰睡眠、浪费消防资源；而漏报的成本可能是建筑焚毁、人员伤亡。因此，我们可能更愿意接受一定的误报以换取极低的漏报率。

此外，**攻击的基础发生率**也会极大影响检测器的实用性。即使一个检测器误报率很低（如0.1%），如果正常请求量极大（如1000万次），而攻击次数很少（如5次），那么检测器仍会产生大量误报（1万次），使得从海量警报中找出真实攻击变得异常困难。这在统计学上被称为**基准率谬误**。

最后，可以通过组合多个检测器来调整性能。**并行组合**（任一检测器报警则总系统报警）通常会降低漏报率，但可能增加误报率。**串联组合**（所有检测器都报警则总系统报警）通常会降低误报率，但可能增加漏报率。

## 检测攻击的策略 🔍

确定了部署位置和衡量标准后，我们来看看检测器内部具体使用哪些策略来判断是否存在攻击。

### 基于签名的检测 ✍️

这种策略维护一个已知攻击模式的数据库（即“签名”或“拒绝列表”）。如果观测到的数据与列表中的任何签名匹配，则触发警报。

例如，检测路径遍历攻击可以查找请求中是否包含 `../` 序列。检测缓冲区溢出攻击可以查找是否包含常见的shellcode字节序列。

*   **优点**：简单直接；对于已知攻击非常有效；社区有共享的签名库。
*   **缺点**：无法检测新的或未知的攻击；攻击者可通过轻微变种绕过签名；可能需要对数据（如URL编码）进行正确解析。

### 基于规范的检测 📋

这种策略与基于签名相反。它定义一个所有“合法”或“允许”的行为列表（即“允许列表”）。任何不在该列表上的行为都会被标记为可疑。

例如，对于文件访问，可以规定只允许字母和数字组成的文件名，那么包含 `../` 的输入就会被拒绝。对于一个询问年龄的程序，可以规定输入只能包含数字0-9。

*   **优点**：能够检测未知攻击，只要其不符合规范；通常误报率较低。
*   **缺点**：需要为每个应用程序手动制定规范，工作量大，难以通用；如果规范制定有遗漏，会产生漏洞。

### 基于异常的检测 🤖

这种策略利用机器学习或统计模型来建立系统“正常”行为的基线。当观测到的行为显著偏离这个基线时，就触发警报。

例如，模型可能通过学习发现，正常的用户输入通常由可打印字符组成，而包含大量非打印字符（如 `\xff`）的输入往往与攻击相关。

*   **优点**：有潜力检测未知攻击。
*   **缺点**：模型可能不准确，导致漏报或误报；如果“异常”行为也可能是合法操作，则误报率高；在实践中难以可靠实施。

### 基于行为的检测 🎯

这种策略不直接分析输入内容，而是监控输入所引发的**系统行为**（如文件操作、进程执行），并寻找攻击证据。

例如，检测路径遍历攻击可以监控程序是否试图读取像 `passwords.txt` 这样的敏感文件。检测缓冲区溢出可以监控是否意外执行了 `exec()` 这类危险函数。

*   **优点**：能够检测未知攻击；某些行为（如执行 `exec()`）是攻击的强信号，误报率低；有时可以借助操作系统已有的监控功能。
*   **缺点**：可能只能在攻击成功后检测；只能检测到成功的攻击；攻击者可能试图掩盖恶意行为。

## 其他相关主题 🧩

最后，我们简要浏览一些与入侵检测相关的扩展主题。

*   **漏洞扫描**：在系统上线前，主动使用工具或人工方式模拟攻击，以发现并修复潜在漏洞。这是一种主动的安全加固措施。
*   **蜜罐**：部署一些看似有价值、实则无用的假系统（“蜜罐”），用于诱捕攻击者。任何对蜜罐的访问都极有可能是恶意行为，从而触发警报。栈保护金丝雀（Stack Canary）就是一个蜜罐思想的例子。
*   **事件响应与取证**：攻击发生后，需要进行调查（取证）以了解事件全貌、恢复系统、修补漏洞并防止再次发生。这包括分析日志、备份恢复等一系列工作。
*   **入侵防御系统**：将检测与预防功能结合的系统。它不仅能发现攻击，还能实时阻止攻击（例如丢弃恶意数据包）。但这要求系统具备干预能力，并且需要谨慎对待误报，因为错误的阻止会干扰正常用户。

## 总结 📚

本节课中我们一起学习了入侵检测的各个方面。

1.  **动机**：我们以**路径遍历攻击**为例，说明即使有防御措施，也需要能够检测攻击。
2.  **部署位置**：探讨了三种主要位置：**网络入侵检测系统**（成本低但存在解读不一致）、**基于主机的入侵检测系统**（准确但部署复杂）和**日志分析**（成本低但只能事后检测）。
3.  **性能衡量**：引入了**误报率**和**漏报率**两个关键指标。指出需要在两者间权衡，且检测器效果受**攻击基础发生率**和**错误成本**影响。
4.  **检测策略**：介绍了四种核心策略：**基于签名**（匹配已知模式）、**基于规范**（匹配允许列表）、**基于异常**（偏离正常模型）和**基于行为**（监控系统行为）。每种策略各有其优缺点和适用场景。
5.  **扩展主题**：简要了解了漏洞扫描、蜜罐、事件响应和入侵防御系统等相关概念。
