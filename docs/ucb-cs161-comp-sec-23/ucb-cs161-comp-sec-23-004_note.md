## 格式化字符串漏洞

上一节我们回顾了缓冲区溢出等基础内存安全问题，本节中我们来看看一种不同风格的漏洞：格式化字符串漏洞。这种漏洞要求我们非常熟悉 `printf` 函数的工作原理。

你可能还记得，`printf` 函数接受可变数量的参数。它如何知道接收了多少个参数呢？这依赖于你提供的**格式化字符串**与后续传入参数之间的映射关系。

例如，一个 `printf` 调用如下：
```c
printf("%s costs %d", fruit, price);
```
第一个参数是特殊的**格式化字符串**。字符串中的 `%` 符号是特殊的格式说明符。`printf` 会按顺序将格式说明符与后续的参数配对：第一个 `%s` 对应 `fruit`，第二个 `%d` 对应 `price`。

`printf` 的一般工作流程是：读取格式化字符串，逐个字符输出，遇到 `%` 格式说明符时，就从栈上找到对应的参数，将其值替换进去。

### 参数不匹配会怎样？
但我们是安全课程，关注的是非预期行为。如果我们传入的参数数量与格式化字符串中的 `%` 说明符数量不匹配，会发生什么？

以下是一个正常的调用：
```c
printf("%d", 123); // 输出：123
```
其栈布局大致如下（函数调用时参数逆序压栈）：
*   压入参数 `123`
*   压入格式化字符串 `"%d"` 的地址
*   调用 `printf`，创建其栈帧

`printf` 会读取格式化字符串，看到 `%d`，然后到栈上（格式化字符串地址之上）寻找第一个未使用的参数（即 `123`），将其作为整数打印。

现在，考虑一个不匹配的调用：
```c
int secret = 42;
printf("%d"); // 只传了一个参数（格式化字符串）
```
栈布局与之前类似，但少压入了一个参数 `123`。`printf` 的工作逻辑不变：它看到 `%d`，仍然会到栈上格式化字符串地址之上的位置去寻找“第一个参数”。然而，那里并没有我们传入的 `123`，取而代之的可能是局部变量 `secret` 或其他栈上的数据。于是，本应保密的 `secret` 值 `42` 就被意外打印了出来。

**核心问题**：当格式化字符串中的格式说明符多于提供的参数时，`printf` 会盲目地从栈上读取数据来填充这些说明符，导致信息泄露。

### 漏洞代码模式
一个典型的易受攻击代码模式如下：
```c
char buff[64];
fgets(buff, sizeof(buff), stdin); // 安全地读入最多63个字符
printf(buff); // 危险！将用户输入直接作为格式化字符串
```
`fgets` 限制了写入长度，防止了缓冲区溢出。但问题出在 `printf(buff)`。如果攻击者控制 `buff` 的内容，他们就可以注入 `%` 格式说明符。

**漏洞根源**：程序员将用户控制的字符串直接作为 `printf` 的**第一个参数**（即格式化字符串）。这个参数是特殊的，`printf` 会解析其中的 `%` 符号。

### 攻击者能做什么？
攻击者可以通过输入特定的字符串进行利用：

以下是攻击者可能输入的内容及其效果：

*   **`%d`**：`printf` 会从栈上读取4字节数据，并将其作为整数打印，泄露栈内存。
*   **`%s`**：`printf` 会从栈上读取4字节数据，将其**解释为一个指针（地址）**，然后前往该地址，将其后的内存作为字符串（直到遇到空字节`\0`）打印出来。这可能导致从任意内存地址读取数据。
*   **`%x %x %x`**：多次泄露栈上数据。
*   **`%n`**：这是一个特殊的格式说明符。它同样从栈上读取一个地址，但**不打印内容，而是将截至目前已输出的字符数写入该地址**。这实现了**内存写**操作！

`%n` 说明符示例：
```c
int val;
printf("Hello%n", &val); // val 将被赋值为 5（"Hello"的长度）
```
在参数不匹配的情况下，攻击者可以利用 `%n` 将数据写入栈上的任意地址（取决于栈布局），从而可能修改关键变量或控制流。

### 利用格式化字符串进行任意写
让我们看一个更复杂的例子，目标是向特定地址 `0xdeadbeef` 写入值 `100`。

假设存在以下易受攻击函数：
```c
void vulnerable() {
    char buff[16];
    fgets(buff, sizeof(buff), stdin); // 假设这里实际限制为16，防止溢出
    printf(buff); // 漏洞点
}
```
攻击者需要精心构造输入，使得当 `printf` 执行时满足两个条件：
1.  当遇到 `%n` 时，它从栈上取出的“参数”正好是目标地址 `0xdeadbeef`。
2.  在遇到 `%n` 时，已经输出的字符数正好是 `100`。

这需要精确地“调校”输入字符串，包括：
*   使用 `%c`、`%d`、`%s` 等消耗栈上的“参数”，将 `%n` “对准”到存储 `0xdeadbeef` 的栈位置。
*   使用形如 `%94c` 的格式说明符来精确控制输出的字符数（`%94c` 会输出一个字符，并用空格填充至总宽度94），从而凑够总长度100。

一个构造的输入可能如下（概念示意）：
```
"\xef\xbe\xad\xde%94c%c%c%n"
```
这个字符串的构造过程涉及对栈布局的精确计算，是项目一中需要掌握的技巧。

### 如何防御？
防御的关键很简单：**永远不要让用户控制 `printf` 的第一个参数（格式化字符串）**。
应该使用：
```c
printf("%s", user_input); // 安全：格式化字符串是常量
```
而不是：
```c
printf(user_input); // 危险！
```
确保格式化字符串是程序内定义的常量，攻击者就无法注入 `%` 说明符。

## 差一错误漏洞

上一节我们介绍了利用 `printf` 的复杂漏洞，本节中我们来看看另一种精巧的利用方式：“差一错误”。这种漏洞允许攻击者仅修改一个字节，就能最终劫持程序执行流。

考虑以下代码：
```c
void vulnerable() {
    char name[20];
    // 假设一个函数允许写入21字节
    read_into(name, 21); // 多写1个字节！
}

int main() {
    vulnerable();
    return 0;
}
```
`name` 数组大小为20字节，但函数允许写入21字节。这意味着攻击者可以覆盖 `name` 数组之后**紧邻的1个字节**。

### 栈布局与攻击目标
画出 `vulnerable` 函数调用时的栈帧：
```
[高地址]
... (调用者栈帧) ...
main 函数的返回地址 (RIP)
main 函数的栈帧指针 (SFP)
------------------ vulnerable 栈帧开始 ------------------
vulnerable 的返回地址 (RIP)
vulnerable 的旧栈帧指针 (SFP) <- 攻击者可以修改其最低1字节
name[20] 数组
------------------ vulnerable 栈帧结束 ------------------
[低地址]
```
攻击者能完全控制 `name[20]` 的20字节，并能覆盖 `SFP` 的最低有效位1字节。

### 利用原理
1.  **修改SFP**：攻击者覆盖 `SFP` 的最低字节，使其指向一个更低的地址（例如，指向 `name` 数组内部的某个位置）。
2.  **混淆程序**：当 `vulnerable` 函数返回时，它会执行标准序言操作：
    *   `pop ebp`：将栈上保存的 `SFP` 值恢复至 `EBP` 寄存器。由于我们修改了它，现在 `EBP` 指向一个错误的位置（`name` 数组内部）。
    *   `ret`：从栈上弹出 `RIP` 并跳转。这个 `RIP` 我们**没有直接修改**，所以程序正常返回到 `main` 函数。
3.  **关键效果**：虽然执行流回到了 `main`，但 `EBP` 寄存器现在指向了被攻击者控制的 `name` 数组区域。在内存中，`EBP` 指向的位置被程序**误解**为是前一个栈帧（`main` 的栈帧）的 `SFP` 所在处。
4.  **第二次返回**：当 `main` 函数随后返回时：
    *   它根据当前 `EBP` 的值来定位自己的栈帧（现在这个定位是错误的）。
    *   它从“被误认为的栈帧”中取出“返回地址”（实际上是 `name` 数组中攻击者放置的数据，例如 `shellcode` 的地址 `0xdeadbeef`）。
    *   将这个值弹出到 `EIP`，程序随即跳转到攻击者指定的地址（`shellcode`）执行。

**核心思想**：通过修改一个字节（`SFP` 的低位），改变了 `EBP` 寄存器的值。这个错误的 `EBP` 在后续函数返回时，导致程序对栈帧布局产生误解，从而从攻击者控制的内存区域（`name` 数组）中读取了伪造的返回地址。

### 总结
差一错误利用展示了即使是最微小的内存破坏（一个字节），在特定的程序状态和栈布局下，也可能被串联起来形成完整的代码执行漏洞。它强调了边界检查必须**精确无误**的重要性。

## 本节课总结
本节课我们一起学习了两种复杂的内存安全漏洞利用技术：
1.  **格式化字符串漏洞**：源于将用户输入直接作为 `printf` 的格式化字符串，使得攻击者可以泄露内存或实现任意内存写，特别是利用 `%n` 说明符。
2.  **差一错误漏洞**：通过溢出数组一个字节来修改相邻的栈帧指针（`SFP`），利用后续函数返回时的栈帧恢复机制，误导程序执行攻击者注入的代码。
